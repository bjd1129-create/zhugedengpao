# 问题诊断：系统化根因分析

## 适用场景

- 研究产出不符合预期
- 工具调用失败（web_search/web_fetch/browser）
- Agent响应异常
- 工作流程阻塞
- 数据不一致

## 诊断流程

### Phase 1: 信息收集

1. **理解症状**
   - 具体发生了什么？
   - 期望发生什么？
   - 什么时候开始的？
   - 最近有什么变化？

2. **收集上下文**
   - 查看相关文件内容
   - 检查工具返回的错误信息
   - 查看 memory/ 当日日志

3. **复现问题**
   - 如果可能，尝试重现问题
   - 记录复现步骤

### Phase 2: 根因分析

1. **形成假设**
   - 什么可能导致这个症状？
   - 考虑：工具限制、配置错误、上下文丢失、资源耗尽

2. **验证假设**
   - 通过日志或测试验证每个假设
   - 追踪调用链找到确切问题点

3. **定位根因**
   - 找到导致问题的最根本原因
   - 不是修复症状，而是修复根因

### Phase 3: 修复验证

1. **实施修复**
   - 做最小必要修改
   - 确保修复针对根因而非表象

2. **验证修复**
   - 确认问题已解决
   - 检查是否引入新问题

3. **记录教训**
   - 如果是值得记录的问题，写入 `.learnings/ERRORS.md`

## 按问题类型诊断

### "工具调用失败"
- 获取完整错误信息
- 检查API配置/权限/限流
- 尝试替代方案

### "研究产出空洞"
- 检查信息来源是否可靠
- 核对原始任务要求
- 评估是否需要补充调研

### "Agent响应异常"
- 检查上下文是否被截断
- 查看是否触发了内容过滤
- 评估模型是否适合任务类型

### "工作流阻塞"
- 定位阻塞点
- 分析依赖关系
- 找到绕过或解决路径

## 输出格式

```
## 问题诊断报告

### 症状
[具体描述]

### 根因
[最根本的原因]

### 修复方案
[具体修改/操作]

### 验证结果
[修复后的状态]

### 教训记录
[是否写入 .learnings/ERRORS.md]
```
