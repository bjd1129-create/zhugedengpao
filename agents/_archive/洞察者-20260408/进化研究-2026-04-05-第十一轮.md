### 1. OpenClaw失败模式深度分析（BetterClaw 2026-04）

**核心发现**：大多数OpenClaw Agent在两周内死掉。能存活超过两周的只有特定模式。BetterClaw通过观察数百个部署案例，总结出7条规律。

**最常见死亡模式**：

| 死亡模式 | 原因 | 后果 |
|---------|------|------|
| API账单爆炸 | 一个模型处理所有任务（包括heartbeat） | $50-150/月 → 放弃项目 |
| 记忆溢出幻觉 | 无结构的长期记忆积累 | 第10次对话后开始胡说 |
| 恶意技能入侵 | 安装未审核的ClawHub技能 | 安全事件 |
| 更新后崩溃 | 配置未版本控制 | 无法回滚 |
| Runaway Loop | 工具报错→重试→再报错循环 | 一小时烧$50-100 |

**关键数据**：
- 稳定Agent平均寿命：47天+
- 单Agent MTBF：847小时
- 多Agent编排MTBF：312小时（更低，复杂性代价）
- 最稳定的用户：每个操作都有hard spending cap

### 2. OpenClaw-RL 深度技术解析（arXiv:2603.10165）

**论文信息**：Princeton团队，2026-03-10，arXiv:2603.10165

**核心观察**：
> 每个Agent交互的每个动作之后，都必然产生"下一状态信号"——用户回复、工具输出、终端/GUI状态变化。这是普遍存在的，无需额外标注，是天然的训练数据来源。

**两种信号提取方式**：

```
评估性信号（Evaluative Signals）
├── 来源：PRM Judge（Process Reward Model）
├── 形式：标量奖励（Scalar Reward）
└── 回答："这个动作做得好不好？"

指令性信号（Directive Signals）
├── 来源：OPD（Hindsight-Guided On-Policy Distillation）
├── 形式：Token级方向性监督
└── 回答："这个动作应该怎么做不同？"
```

**OPD（On-Policy Distillation）工作流**：
1. 从下一状态提取文本提示（textual hints）
2. 构建增强教师上下文（enhanced teacher context）
3. 提供token级方向性监督（比标量奖励信息量更大）

**异步设计三组件**：
```
┌─────────────────┐     ┌──────────────┐     ┌──────────────┐
│  模型服务（实时）│ ←→  │  PRM评判      │ ←→  │  训练器（更新）│
│  处理用户请求    │     │  评估进行中   │     │  策略梯度更新  │
└─────────────────┘     │  交互         │     └──────────────┘
                        └──────────────┘
          零协调开销（zero coordination overhead between them）
```

**实践意义**：
- 对话本身就是训练数据——不需要传统RLHF流水线
- 用户重新查询（re-queries）、纠正（corrections）、明确反馈（explicit feedback）都是训练信号
- 同一套基础设施支持：对话、终端执行、GUI交互、SWE任务、工具调用

### 3. self-evolve插件架构深度解析（longmans）

**技术架构**：基于Q学习（Q-Learning）的自我进化系统

**两个核心Hook**：

```
before_prompt_build hook：
├── 管理pending task状态（open / waiting_feedback）
├── 检测反馈、意图切换、空闲关闭、TTL关闭、最大轮次关闭
├── 构建embedding并检索候选记忆
└── 如有候选则注入；无候选则保持task pending（bootstrap）

agent_end hook：
├── 捕获assistant响应
└── 将task状态移动到waiting_feedback
```

**Q值学习机制**：
- 每个"情节记忆"（episodic memory）关联一个Q值
- 反馈检测后：评分reward → 检查learning gates → 更新Q值
- 新记忆写入：本地清洗 → LLM摘要 → 二次清洗 → 追加情节记忆三元组
- 可选：按request_key_id远程ingest

**Learning Gates（默认阈值）**：
```python
runtime.observeTurns = 0       # 观察轮数
runtime.minAbsReward = 0.15    # 最小绝对reward
runtime.minRewardConfidence = X # 最小reward置信度
```

**反馈质量原则**：
- 明确表扬/批评 > 模糊的"ok"
- Positive reinforcement效果更好
- Explicit feedback是首选信号

**推荐升级**：OpenClaw 2026.3.2+（旧版本可能缺少hook context，无法可靠捕获工具调用trace）

### 4. Self-Improving Agent Skill架构（xiucheng）

**定位**：自我改进Agent系统，分析对话质量、识别改进机会、持续优化响应策略

**Security扫描结论**（Benign, High Confidence）：
- 唯一系统访问：workspace下的文件读写（improvement_log.md, SOUL.md）
- 无网络调用、无凭证请求
- 无持久后台服务
- MIT-0 License（可自由使用、修改、分发）

**功能组件**：
- analyze_conversation：分析对话文本质量
- log：写入improvement_log.md
- report：生成报告
- 自动会话后分析（需agent传递对话内容）
- 改进日志晋升路径：improvement_log → daily memory → CLAUDE.md/AGENTS.md

**与self-evolve的关系**：互补
- self-evolve = 算法学习（Q值驱动）
- Self-Improving Agent = 结构化日志（模板驱动）

### 5. 构建自我改进Agent完整工作流（blog.exit.st）

**四阶段循环**：

```
Phase 1: 检测（Detection）
├── 任务失败：无法完成或完成质量差
├── 重复模式：同类请求反复出现但无专门处理器
├── 用户反馈：明确纠正或不满表达
└── 效率指标：耗时过长或步骤过多

        ↓

Phase 2: 分析（Analysis）
├── 缺什么能力？
├── 解决方案形态？
├── ClawHub已有相关技能？
└── 修改现有技能 or 创建新技能？

        ↓

Phase 3: 生成（Generation）
├── 编写新SKILL.md（含指令）
├── 创建支撑脚本或模板
├── 用触发场景测试技能
└── 验证现有能力未被破坏

        ↓

Phase 4: 集成（Integration）
├── 技能文件放入正确目录
├── 简短验证运行确认有效
├── 改进日志记录供人工审查
└── Agent立即开始使用新能力
```

**性能监控数据模型**（performance-log.json）：
```json
{
  "task": "客户问题回复",
  "status": "success|failure",
  "duration_ms": 2340,
  "error": "error message if any",
  "feedback": "user correction if any"
}
```

**改进触发条件**（建议阈值）：
- 同一类型失败 ≥ 3次/7天
- 任务耗时超过预期 2倍
- 用户明确纠正 ≥ 1次

## 二、运营稳定性最佳实践（新补充细节）

### BetterClaw 7条稳定规则（详细版）

**规则1：模型路由**
```
任务类型          → 推荐模型           → 成本对比
Heartbeat/轮询   → DeepSeek Haiku     → $0.28/$0.42/M
简单对话         → Gemini Flash      → ~$0.10/M  
复杂多步推理     → Claude Opus/GPT-4o → $15/$30/M
文件摘要         → Sonnet 4           → $3/M
```
⚠️ 错误：所有任务用Opus → $80-150/月  
✅ 正确：路由模型 → $14-25/月（节省70-80%）

**规则2：消费上限**
- 月度上限 = 期望月费的2-3倍
- 单次响应maxIterations = 10-15（防止长链调用）
- 立即设置！runaway loop可以在1小时内烧掉$50-100

**规则3：结构化SOUL.md**
❌ 差的SOUL.md：`Be helpful and friendly.`（太模糊，50次对话后Agent行为漂移）
✅ 好的SOUL.md包含：
- 错误状态行为定义
- 对话边界规则
- 速率限制语言
- 话题限制规则
- 特定场景的标准操作程序

**规则4：安全基线**
```
Gateway绑定：127.0.0.1（不是0.0.0.0）
SSH认证：Key-based（非密码）
防火墙：UFW默认deny
技能审核：安装前必须看源码（20%的ClawHub技能质量低或有问题）
CVE-2026-25253：RCE漏洞，CVSS 8.8，必须升级到2026.3+
```

**规则5：ContextEngine（OpenClaw 2026.3+）**
- 50轮对话context减少30%（token省30%）
- /btw 命令：长任务中途问无关问题不破坏主任务上下文

**规则6：Secrets Workflow**
```bash
~/.openclaw/secrets/  # 加密本地存储
.gitignore:           # 自动排除
```
永远不要把secrets放进config或AGENTS.md

**规则7：Human-in-the-Loop**
- 敏感操作暂停："你要我删除这个，对吗？"
- 减少"手滑"事故
