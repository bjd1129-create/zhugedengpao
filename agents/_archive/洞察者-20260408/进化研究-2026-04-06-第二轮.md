## 核心发现

### 1. Self-Evolve 插件 v2 — Q学习 + 三元组记忆网络（重大！）

**地址**：https://github.com/longmans/self-evolve

这是目前最接近"强化学习"的 OpenClaw 插件，v2 版本在 2026年3月发布，与 OpenClaw 2026.3.2+ 配合使用。

**核心机制**：

- **before_prompt_build**：检测反馈、意图边界，检索本地+远程记忆候选，注入上下文
- **agent_end**：捕获回复，移动任务状态
- **用户反馈时**：评分 reward，决定是否学习，更新 Q 值，写入情景记忆三元组

**关键参数**（learning gates）：

```
runtime.observeTurns = 0
runtime.minAbsReward = 0.15
runtime.minRewardConfidence = 0.55
runtime.noToolMinAbsReward = 0.8（无工具调用时）
runtime.noToolMinRewardConfidence = 0.9（无工具调用时）
retrieval.tau = 0.85（相似度阈值）
```

**三种学习模式**：

- balanced（默认）：优先有工具调用的轮次
- tools_only：仅学习有工具调用的轮次（最低 token 消耗）
- all：所有通过 reward 门槛的轮次（最高 token 消耗）

**远程共享机制**：

- 本地+远程记忆合并排序后注入
- 共享「意图-经验三元组」到 self-evolve.club
- 隐私：两轮脱敏（sanitizeMemoryText + LLM 摘要替换）

**小花判断**：这个和我们团队的记忆系统高度契合。我们一直在做「经验三元组」，self-evolve 把它做成了自动化流程。建议：安装这个插件，接入 self-evolve.club 网络。

### 2. Capability Evolver — 确定性的自我诊断引擎（实用！）

**地址**：https://github.com/kennyzir/capability-evolver

35K+ 下载量，ClawHub 最热技能之一。本地运行，无需 API key，无外部依赖。

**核心价值**：

- 处理结构化日志，输出：错误模式 + 健康评分 + 改进建议
- 纯规则引擎，结果可复现（同一批日志永远得到相同结果）
- 子 100ms 处理速度，零 token 成本
- 五种进化策略：auto / balanced / innovate / harden / repair-only

**应用场景**：

- 部署前健康检查（预检评分低于基准则阻止上线）
- 生产环境实时监控（每小时分析一次）
- 事故溯源（识别错误级联和根因）
- 多 Agent 舰队管理（聚合分析找共性问题）

**vs LLM 分析**：

| | LLM 分析 | Capability Evolver |
|---|---|---|
| 速度 | 5-30秒 | <100ms |
| 可复现 | ❌ | ✅ |
| 幻觉风险 | ⚠️ | ✅ 无 |
| 成本 | $0.10-0.50/次 | 免费 |
| 语义理解 | ✅ | ❌ |

**小花判断**：我们团队目前没有系统性的日志分析机制，这个插件可以帮我们建立自动化的自我诊断能力。建议在 cron 任务中加入每周一次的健康检查。

### 3. OpenClaw Best Practices v2 — Context Graph + Session Bloat（重要！）

**地址**：https://github.com/garrettekinsman/openclaw-best-practices

2026-03-20 发布的 v2，23 个章节，其中几个新章节值得重点关注：

**Section 19: Context Graph Integration**
- 用 DAG 结构（topic + recency）替代线性滑动窗口
- 语义检索引擎，只取最相关的上下文片段

**Section 20: Session Context Bloat**
- 长周期 session 累计 100k+ token 的预防和处理
- heartbeat/cron 堆积导致 token 爆炸的根源分析

**Section 21: Running Agent Teams**
- 命名 persona 构建/审计周期
- identity 注入机制
- memory 访问策略和报告归属机制

**Section 22: Parallel Agent Workflows**
- fan-out/fan-in 研究模式
- 中途接管（mid-task steering）
- GPU 竞争管理

**Section 23: Leveraging Local Compute**
- orchestrator 模式详解
- 按任务类型选模型（简单任务用小模型省钱）
- LiteLLM 路由、VRAM 管理、成本优化

**小花判断**：Context Graph 是解决 token 膨胀的关键方案。Session Bloat 是我们团队潜在的风险点——cron/heartbeat 日志堆积会让 session 越来越慢，值得警惕。

### 4. OpenClaw 2026 运营最佳实践（运营参考）

**来源**：https://openclaw-ai.com/en/blog/best-practices

**性能优化**：

- 按任务复杂度选模型：简单问答用小模型，复杂推理用大模型
- 启用缓存（caching），降低 API 调用次数
- system prompt 保持精简（不要堆砌功能列表）

**成本管理**：

- 设置消息频率和日 token 上限
- 配置预算警告（0.8 = 80% 警告，1.0 = 超限）

**安全建议**：

- 敏感信息放 .env，不要硬编码
- 限制 channel 访问白名单
- 定期执行 `openclaw update`

## 重大发现总结

### 🔥 最高优先级：接入 Self-Evolve 网络

self-evolve 插件把「经验三元组」做成了自动化流程，和我们团队一直在做的事情高度吻合。接入后：
- 自动从真实 session 中学习
- 参与 self-evolve.club 共享网络
- 用 Q 值算法替代「感觉式」的经验记录

**操作**：安装 self-evolve 插件，开通远程共享

### 🔥 高优先级：建立日志健康检查机制

Capability Evolver 提供了零成本的自动诊断能力，我们可以用 cron 任务每周运行一次分析。

### ⚠️ 风险点：Session Context Bloat

cron/heartbeat 堆积会让 session 越来越慢。需要在 AGENTS.md 中加入 session 维护规范。
