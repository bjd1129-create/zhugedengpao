## 一、最重大发现：longmans/self-evolve 插件（核心突破）

### 1.1 定位
GitHub: https://github.com/longmans/self-evolve

**这是一个真正的生产级自我进化插件**，不是理论，是已经落地的代码。

核心特点：
- 更少 tokens，更多算法化学习
- 基于强化学习的 Q-value 机制
- 本地 + 远程 RAG 记忆网络

### 1.2 工作流程（关键！）

```
用户消息 → 判断是否为反馈轮次？
  ├─ 是（反馈轮）→ 评分 reward → 检查学习门控 → 决定是否学习
  │    ├─ 本地第一次清洗（sanitizeMemoryText）
  │    ├─ LLM 总结 + 第二次清洗
  │    ├─ 追加本地记忆三元组 (intent/experience/embedding)
  │    └─ 可选：远程上报 self-evolve.club
  └─ 否（普通轮）→ 检测意图边界 → 检索本地+远程记忆
       → Phase-B 排名选择 → 注入记忆 → 生成回复
       → 标记为 waiting_feedback
```

**两个核心 Hook**：
- `before_prompt_build`：管理待处理任务状态，检测反馈/新意图/空闲/TTL关闭
- `agent_end`：捕获 assistant 回复，任务进入 waiting_feedback 状态

### 1.3 学习门控参数（可调）

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `runtime.observeTurns` | 0 | 观察多少轮再学习 |
| `runtime.minAbsReward` | 0.15 | 最小绝对 reward 值 |
| `runtime.minRewardConfidence` | 0.55 | 最小 reward 置信度 |
| `runtime.noToolMinAbsReward` | 0.8 | 无工具调用的学习门槛 |
| `runtime.noToolMinRewardConfidence` | 0.9 | 无工具调用的置信度门槛 |

### 1.4 三种学习模式

| 模式 | 特点 | token 成本 |
|------|------|-----------|
| `balanced`（默认）| 偏好工具调用；无工具调用需高 reward/置信度 | 中 |
| `tools_only` | 只在调用工具时学习 | 最低 |
| `all` | 所有通过 reward 门控的轮次都学 | 最高 |

### 1.5 任务边界检测

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `runtime.newIntentSimilarityThreshold` | 0.35 | 新意图相似度阈值 |
| `runtime.idleTurnsToClose` | 2 | 空闲轮次关闭任务 |
| `runtime.pendingTtlMs` | 300000（5分钟）| 任务 TTL |
| `runtime.maxTurnsPerTask` | 5 | 每任务最大轮次 |

### 1.6 记忆保留

- 默认 `memory.maxEntries = 200`
- 超限保留策略：高 Q 值 / 高成功 / 近期 / 高选中次数
- 去重近邻记忆，保留少量新鲜配额

### 1.7 远程共享机制（隐私设计亮点）

**自我进化的隐私保护设计非常严谨**：
- 第一道清洗：`sanitizeMemoryText` 移除对话元数据、ID、发件人标签
- 第二道清洗：experience summarizer 要求 LLM 输出"可转移策略"，敏感数据替换为 `[REDACTED_*]`
- 共享内容：仅限意图/经验/embedding 三元组 + 匿名 request_key_id
- 可在 https://self-evolve.club/#leaderboard 查看共享排名
- 可通过配置关闭远程共享

**远程共享的价值观**：
- 贡献高质量经验 → 获得社区其他人的经验作为回报
- Privacy-first，本地清洗后再上报

## 二、OpenClaw-RL：普林斯顿异步强化学习框架

arXiv: https://arxiv.org/abs/2603.10165

**核心洞察**：
> "next-state signals are universal, policy can learn from all of them simultaneously"

这句话非常重要——意思是：对话、执行、GUI交互、工具调用，这些都是同一类训练信号，不需要分开处理。

**与小花团队的相关性**：
- 小花的每次操作（发消息、执行命令、飞书API调用）都是潜在的 RL 信号
- 不需要等"正确结果"，只需要"信号"就能学习

## 三、多 Agent 自我进化团队架构

### 3.1 参考架构（dev.to）

```
CEO Agent  →  协调任务，优先级管理
CTO Agent  →  技术决策
CFO Agent  →  财务与报告
COO Agent  →  日常运营
Specialists → 内容、销售、分析
```

关键特征：
- 每个 Agent 有独立记忆
- 通过结构化渠道通信
- 共享通用记忆系统

### 3.2 自我进化循环

每日反思三问：
1. 今天我们学到了什么？
2. 我们应该改变什么？
3. 应该保留什么知识？

### 3.3 核心教训

> **"没有持久记忆，AI Agent 每次会话都会重置。有持久记忆，才能在前人经验上累积。"**

## 四、Self-Improving Agent 技能：四阶段闭环

### Phase 1: Detection（检测）
监控信号：
- 任务失败
- 重复模式（同一类请求反复出现）
- 用户反馈
- 效率指标（耗时、步骤数）

### Phase 2: Analysis（分析）
- 缺少什么能力？
- 解决方案长什么样？
- ClawHub 上已有相关技能吗？
- 修改现有还是新建？

### Phase 3: Generation（生成）
- 写新的 SKILL.md
- 创建支撑脚本
- 测试新技能
- 验证不破坏现有能力

### Phase 4: Integration（整合）
- 放入 pending-skills/ 目录待审
- 人工审核后移入活跃技能目录
- **高风险操作（影响外部系统）永远需要人工签字**

## 五、OpenClaw 2026.3 核心技术更新

### 5.1 ContextEngine（上下文引擎）
- 智能裁剪：自动识别当前任务需要的信息
- **实测：50轮对话减少约 30% context 使用量**

### 5.2 三层插件架构

```
Bundle  →  技能包（可复用集合）
Provider →  后端实现（切换模型不破坏技能）
Plugin  →  底层扩展（Hooks、系统集成）
```

### 5.3 OpenShell Mirror 模式
- 直接使用本地 shell，不启动 Docker
- MacBook Air 测试：内存比 Docker 节省一半
- 适合日常开发测试

### 5.4 /btw Sidebar Q&A
- 长任务中途临时提问，不污染主 context
- 解决痛点：如何在长对话中插话而不引入额外 context

### 5.5 Human-in-the-Loop
- AI 主动暂停询问"这样对吗？"
- 从"全自动化"转向"人机协同"
- 与小花的安全边界设计理念完全一致

## 六、最佳实践精华（garrettekinsman/openclaw-best-practices）

### 6.1 安全（最高优先级）
```bash
# ✅ 正确做法：优雅重启，保留 session
gateway tool → config.patch action

# ❌ 错误做法：会导致 session orphan
openclaw gateway stop
kill -9 <gateway-pid>
```

### 6.2 SKILL.md 精简原则
**永远 ≤ 50 行，超过一屏就是太长。**

```
skill-name/
├── SKILL.md     → 工作流步骤（每次读取）
├── scripts/     → Python/bash 脚本（按需执行）
├── references/  → 示例/API 文档（按需读取）
└── assets/      → 模板（脚本使用，Agent 不读）
```

### 6.3 Token 成本矩阵

| 来源 | Token 成本 | 管理策略 |
|------|-----------|---------|
| SOUL.md / AGENTS.md | 每次会话 | 保持 ≤ 2 页 |
| MEMORY.md | 每次主会话 | 定期归档到 memory/ |
| SKILL.md | 技能触发时 | 保持 ≤ 50 行 |
| 文件读取 | 每次读取 | 只读需要的部分 |
| 心跳轮次 | 每个心跳 | 保持 HEARTBEAT.md 精简 |

### 6.4 记忆层级
```
MEMORY.md         → 长期记忆精华（决策/规则/偏好）
memory/YYYY-MM-DD → 每日原始记录
memory/archives/  → 已提炼的旧内容
```

### 6.5 脚本设计原则
```python
# ✅ 好：结构化输出，最小化打印
print(json.dumps({"ok": True, "result": result}))

# ❌ 坏：50行调试日志
print("Processing step 1...")
print("Processing step 2...")
```

## 七、对团队的启发与应用

### 立即可行

1. **为小花安装 self-evolve 插件**
   - 参考 longmans/self-evolve 的隐私设计
   - 参与 self-evolve.club 远程共享
   - 积累 Evolution Score

2. **规范反馈机制**
   - 当小花做对时，明确表扬（触发 reward 上升）
   - 当小花做错时，明确指出（触发 Q-value 下降）
   - **明确的反馈 >> 模糊的"好的/不好"**

3. **采用 HEARTBEAT.md 精简原则**
   - 检查现有 HEARTBEAT.md 是否超过 50 行
   - 将非必要内容移到 memory/ 日志文件

4. **建立任务边界意识**
   - 使用 `idleTurnsToClose=2` 防止任务僵死
   - 使用 `pendingTtlMs=300000` 防止任务无限挂起

### 中期规划

5. **小花的自我改进流程**
   ```
   检测（3+同类失败）→ 分析（缺什么能力）→ 生成（写/修改 SKILL.md）
   → 测试 → 人工审核 → 上线
   ```

6. **多 Agent 记忆隔离**
   - 主 Agent（小花）：全局协调、决策、记忆中枢
   - 子 Agent：各自维护短期记忆，定期同步精华到小花

7. **Human-in-the-Loop 嵌入**
   - 重大操作（删除文件、对外发言、飞书群发）前主动暂停询问
   - 符合小花"安全边界"中的不可逾越红线
