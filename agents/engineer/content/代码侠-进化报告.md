## 三、改进计划（Session 2 行动清单）

### 3.1 立即纠正：Session 2 只做纯技术任务

**Session 2 任务清单（5 项，每项均可独立验证）：**

1. **[ ] sitemap.xml 审计**
   - 抓取所有 URL，验证 HTTP 状态码
   - 确认无 404 链接
   - 确认重要页面都在 sitemap 里（首页、articles、diary 等）
   - 输出：`content/SEO审计报告-sitemap.md`

2. **[ ] robots.txt 补充**
   - 当前 robots.txt 只有 `Allow: /` + Sitemap 行
   - 建议添加：User-agent 限制（若需要）、Crawl-delay（若需要）
   - 验证 sitemap.xml 在其中

3. **[ ] SEO meta 抽检（10 个页面）**
   - 检查 og:title / og:description / og:image / canonical
   - 记录每个页面的 meta 完整度
   - 输出：`content/SEO审计报告-meta.md`

4. **[ ] 结构化数据添加（首页）**
   - 添加 Organization JSON-LD（站点级别）
   - 添加 Website/SearchAction JSON-LD
   - 参考：schema.org 标准

5. **[ ] Lighthouse 基准测试**
   - 记录首页 Performance / Accessibility / Best Practices / SEO 分数
   - 建立基准线，用于后续优化对比
   - 输出：`content/SEO审计报告-lighthouse.md`

### 3.2 每日复盘机制（小步版）

不再承诺"写完整的未完成一节"，改为：

```
每次会话结束前，执行以下之一：
- 修改了文件 → 记录在 memory/YYYY-MM-DD.md
- 没改文件 → 记录"今天没改文件，原因：XXX"
```

### 3.3 身份边界清单（内控）

**代码侠职责（纯技术）：**
- 部署、域名、CI/CD、Git 配置
- SEO 技术实现（meta、结构化数据、sitemap/robots）
- 性能优化（加载速度、图片懒加载、缓存）
- 页面技术实现（接收设计稿/内容稿后转成 HTML）
- 技术架构选型

**非代码侠职责（需要协作方）：**
- 首页/科学区/日记内容文案
- 页面设计风格、配色方案
- 用户见证/定价文案
- 博客文章撰写

**遇到身份模糊任务时的判断流程：**
1. 这个任务完成后，哪个文件会变？
2. 如果答案是"改的是 .md 文件或内容文案"，→ 不是代码侠的活
3. 如果答案是"改的是 .html / .css / .js / 配置文件"，→ 是代码侠的活

## 五、今日教训总结（3 分钟版本）

1. **计划必须基于能力边界** — 写计划前先问"这是我的活吗"
2. **产出 = 文件变更** — 没有文件变更 = 没有完成
3. **承诺要小** — "写未完成一节"比"完整复盘"更可能执行
4. **技术欠账不能靠身份错位弥补** — sitemap/robots/Lighthouse 是代码侠的基本功，不能因为"更有价值的事"就跳过
