## 🔍 一、Self-Improving Agent — 事实标准（26 万 + 下载）

### 核心机制

**目录结构**：
```
.learnings/
├── ERRORS.md          # 命令失败、工具报错、推理偏差
├── LEARNINGS.md       # 用户纠正、最佳实践、知识缺口
└── FEATURE_REQUESTS.md # 用户提出的新能力需求
```

**自动触发场景**：
| 场景 | 记录位置 | 类别 |
|------|----------|------|
| 命令/操作失败 | ERRORS.md | error |
| 用户纠正你 | LEARNINGS.md | correction |
| 用户想要缺失功能 | FEATURE_REQUESTS.md | feature |
| API/外部工具失败 | ERRORS.md | integration |
| 知识过时/错误 | LEARNINGS.md | knowledge_gap |
| 发现更优方案 | LEARNINGS.md | best_practice |

**自动复盘机制**：
- 每日凌晨 4 点自动运行（资源最空闲时段）
- 检查当天对话记录，提取有价值经验
- 整理并更新到 MEMORY.md
- 记录版本历史

**ID 生成规范**：`TYPE-YYYYMMDD-XXX`
- `LRN-20260407-001`（学习）
- `ERR-20260407-A3F`（错误）
- `FEAT-20260407-002`（功能）

### 晋升规则（关键！）

当学习条目满足以下条件时，应晋升到永久记忆：

| 学习类型 | 晋升目标 | 示例 |
|----------|----------|------|
| 行为模式 | SOUL.md | "简洁，不端水" |
| 工作流改进 | AGENTS.md | "心跳遵循 Execute-Verify-Report 循环" |
| 工具坑点 | TOOLS.md | "Git push 需先配置 auth" |
| 重复模式（≥3 次） | 系统提示 | 写入 prompt 前置规则 |

**晋升判断标准**：
- 跨多个文件/功能适用
- 任何贡献者（人/AI）都应知道
- 防止重复犯错
- 记录项目特定约定

### 安装与配置

```bash

clawhub install self-improving-agent

# 手动安装
git clone https://github.com/pskoett/self-improving-agent.git ~/.openclaw/skills/self-improving-agent

# 初始化目录
mkdir -p ~/.openclaw/workspace/.learnings
```

**Hook 集成（可选）**：
```bash

cp -r hooks/openclaw ~/.openclaw/hooks/self-improvement

# 启用
openclaw hooks enable self-improvement
```

## 🌱 二、AutoSkill — 技能是从经验中"长出来"的

### 核心理念（论文：arXiv:2603.01145）

> "技能不是设计出来的，而是从经验中长出来的"

**来源**：华东师范大学 + 上海人工智能实验室（2026-03）

**双环架构**：
```
右环（技能进化）：交互体验 → 提取 → 维护 → 显式技能
左环（技能增强）：查询重写 → 技能检索 → 上下文注入 → 响应生成
```

**关键创新**：
- 通过**记忆增长**而非模型微调持续改进
- 技能版本管理（如 `professional_text_rewrite` 已迭代到 v0.1.34）
- 从真实交互中自动提炼可复用模式

### 技能生命周期

```
摄取经验 → 提炼技能 → 复用能力
   ↓           ↓           ↓
对话信号    模式识别    技能库调用
行为轨迹    稳定性判断   版本更新
用户反馈    抽象为规则   合并优化
```

**什么会成长为技能**：
- ✅ 反复出现的稳定约束（如"保持专业语气，避免幻觉"）
- ✅ 用户多次强调的工作流程
- ✅ 跨任务复用的表达风格

**什么不会成为技能**：
- ❌ 一次性请求（如"写一篇报告"）
- ❌ 临时性约束
- ❌ 无复用价值的模式

### 真实案例

**案例 1：顶级心理咨询师**
- 非人工编写，从对话中总结
- 要求：温暖、共情、专业语气
- 边界：不给医疗诊断或药物建议

**案例 2：professional_text_rewrite（v0.1.34）**
- 目标：提高流畅度和专业度
- 约束：严格保留原意，不添加额外解释
- 迭代 34 次，通过用户反馈持续优化

### 小花团队应用建议

**适配场景**：
- 文案君的写作风格沉淀（老庄偏好 → 技能）
- 交易员的决策模式（成功交易特征 → 技能）
- 协调官的沟通模板（高效协作模式 → 技能）

**实施步骤**：
1. 安装 AutoSkill skill
2. 配置技能提取触发器（重复 3 次 + 模式）
3. 建立技能审核流程（小花审批后生效）
4. 定期回顾技能库（删除过时技能）

## 🛠️ 五、最佳实践汇总（Power User 200+ 小时经验）

### 架构设计（Tips 1-3）

**Tip 1：先画 Agent 图再动手**
- 标出每个 agent 的输入、输出、决策点
- 标注数据转换位置
- 标记可能需要人工审批的节点
- 识别可能的失败级联点

**Tip 2：单一职责原则**
- 一个 agent 只做一件事
- 如果一个 agent 有 5 个 + 条件分支 → 拆分
- 判断标准：能否用一句话说清它做什么

**Tip 3：Sub-agent 用于并行，不只是分解**
- 独立任务并发执行
- 45 秒串行 → 20 秒并行（真实案例）
- 识别所有可并行的独立任务链

### 模型路由（Tips 4-6）

**三层模型架构**：
| 层级 | 任务类型 | 推荐模型 |
|------|----------|----------|
| 轻量层 | 分类、路由、简单判断 | Haiku/Flash/GPT-4o Mini |
| 中量层 | 结构化提取、摘要、中等推理 | Sonnet/Claude-3.5 |
| 重量层 | 复杂推理、代码、 nuanced 写作 | Opus/Claude-3.7 |

**专用 Router Agent**：
- 统一入口，根据任务描述分配模型层级
- 集中处理模型不可用 fallback
- 变更时只需改一个 agent

**输出缓存**：
- 重复查询缓存 TTL 策略
- 高流量工作流可省 30-50% 成本
- 审计最频繁触发的工作流

### 通知管理（Tips 7-8）

**Telegram/飞书线程分离**：
```
🚨 Errors & Alerts    → 永不静音，立即处理
✅ Completed Runs     → 定期回顾
⏳ Approvals          → 异步行动队列
ℹ️ Info               → 可静音，定时查看
```

**统一消息格式**：
- 定义模板，所有 agent 强制执行
- 减少解析每条消息的认知负担
- 示例：`[AGENT] [STATUS] 摘要 | 详情链接`

### SKILL.md 苗条原则（2026-04 新共识）

**规则**：SKILL.md 永远 ≤ 50 行

**理由**：
- 超过一屏 = 太长 = context 膨胀
- 复杂逻辑放入 `scripts/` 目录
- 按需文档放入 `references/` 目录

**推荐结构**：
```
skills/{skill-name}/
├── SKILL.md          # ≤50 行，步骤清单
├── scripts/          # 自包含脚本，CLI 参数，JSON 输出
├── references/       # 按需文档
└── assets/           # 模板、示例
```
