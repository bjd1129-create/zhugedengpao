## 🧠 二、OpenClaw-RL — 训练任何 Agent 只需对话

### 论文核心（arXiv:2603.10165 | 2026-03-10）

**核心洞察**：

> "每个 Agent 交互都生成下一个状态信号（用户回复、工具输出、终端/GUI 状态变化），但没有任何现有系统将这些信号作为实时在线学习源。"

**OpenClaw-RL 的简单观察**：
- 下一个状态信号是通用的
- 策略可以从所有信号同时学习

### 两种信号类型

**1. 评估信号（Evaluative Signals）**：
- 指示动作执行得多好
- 通过 PRM（Process Reward Model）评判器提取为标量奖励

**2. 指导信号（Directive Signals）**：
- 指示动作应该有什么不同
- 通过 Hindsight-Guided On-Policy Distillation (OPD) 恢复

### 技术创新：Token 级方向优势监督

**比纯标量奖励更丰富**：
- 从下一个状态提取文本提示
- 构建增强教师上下文
- 提供 token 级方向优势监督

### 异步设计（关键！）

```
模型服务实时请求 ←→ PRM 评判器评判中 ←→ 训练器更新策略
     ↓                     ↓                    ↓
  零协调开销            同时进行              同时进行
```

**优势**：
- 模型服务 live 请求，无需暂停等待训练
- PRM 评判 ongoing 交互
- 训练器同时更新策略
- 三个模块完全异步，零协调开销

### 应用场景

| 场景 | 训练信号来源 |
|------|-------------|
| 个人 Agent | 用户重新查询、修正、显式反馈 |
| 终端执行 | 命令输出、退出码 |
| GUI 交互 | 界面状态变化 |
| SWE 任务 | 代码测试结果 |
| 工具调用 | API 返回结果 |

### 对小花团队的启示

**短期**：无需关注 OpenClaw-RL 集成（需要 PRM 评判器基础设施）

**长期**：
- 如果 OpenClaw 官方推出 RL 训练功能，交易团队的订单反馈可以成为天然训练信号
- 用户修正 → 策略改进的自动闭环

## 🔄 三、self-evolve 插件 — 完整机制解析

### 工作原理（来自 GitHub 官方文档）

**核心架构**：

```
用户消息 → 反馈检测？ → 是 → 打分奖励 + 检查学习门控
                      ↓ 否
              检测意图 + 任务边界
                      ↓
              检索本地 + 远程记忆
                      ↓
              Phase-B 排名/选择记忆
                      ↓
              注入记忆 + 生成回复
                      ↓
              设置任务为 waiting_feedback
```

### 三个学习模式

| 模式 | 策略 | Token 成本 |
|------|------|-----------|
| `balanced`（默认） | 偏好工具调用轮次；无工具轮次需高奖励/置信度 | 中 |
| `tools_only` | 仅在调用工具时学习 | 低 |
| `all` | 所有通过奖励门控的轮次都学习 | 高 |

**无工具轮次的平衡模式阈值**：
- `runtime.noToolMinAbsReward = 0.8`
- `runtime.noToolMinRewardConfidence = 0.9`

### 默认学习门控

```yaml
runtime.observeTurns: 0
runtime.minAbsReward: 0.15
runtime.minRewardConfidence: 0.55
runtime.minFeedbackChars: 已移除
```

### 检索门控

```yaml
retrieval.tau: 0.85  # 仅当最佳相似度足够高时才注入记忆
```

### 任务边界默认值

```yaml
runtime.newIntentSimilarityThreshold: 0.35  # 新意图检测
runtime.idleTurnsToClose: 2                  # 空闲轮次关闭
runtime.pendingTtlMs: 300000                  # 5分钟 TTL
runtime.maxTurnsPerTask: 5                    # 每任务最大轮次
```

### 记忆管理

- 默认 `memory.maxEntries: 200`
- 超出时：保留高价值记忆（Q/成功率/新鲜度/选中次数），去重近似重复，保留少量新鲜配额

### 远程共享机制

**隐私设计（双重脱敏）**：
1. **第一次**：sanitizeMemoryText 移除对话元数据、ID、发送者标签
2. **第二次**：经验摘要器要求 LLM 输出可迁移策略 + 替换敏感数据为 `[REDACTED_*]`

**共享内容**：仅限脱敏三元组（intent / experience / embedding）+ request_key_id 匿名归属

**加入方式**：
```bash
openclaw config set plugins.entries.self-evolve.config.remote.enabled true
openclaw config set plugins.entries.self-evolve.config.remote.baseUrl 'https://self-evolve.club/api/v1'
```

**排行榜**：https://self-evolve.club/#leaderboard

### 小花团队适配评估

**安装状态**：✅ 已安装

**当前配置建议**：
```bash
# 使用 tools_only 模式（最低 token 成本）
openclaw config set plugins.entries.self-evolve.config.runtime.learnMode '"tools_only"'

# 远程共享保持开启（贡献社区，同时获取他人经验）
openclaw config set plugins.entries.self-evolve.config.remote.enabled true
```

**与 .learnings/ 的关系**：
- self-evolve：自动从对话中学习，无需人工触发
- .learnings/：Self-Improving Agent 的主动记录机制
- **两者互补**：self-evolve 负责被动经验积累，.learnings/ 负责主动复盘

## 📊 四、Task Flow — v2026.4.2 回归的复杂任务编排

### 核心价值

**解决的问题**：复杂多步骤任务的状态管理、断点续传、优雅取消

### 两种同步模式

| 模式 | 适用场景 | 优势 |
|------|----------|------|
| 托管模式 | 严格顺序执行、状态强一致 | 精确控制、易调试 |
| 镜像模式 | 可并行、容忍最终一致 | 高并发、容错强 |

### 持久化流状态

存储在 SQLite 数据库（v2026.3.31 统一账本设计），包含：
- 当前执行位置
- 已完成步骤及输出
- 待处理步骤队列
- 上下文变量快照
- 错误与重试计数

**断点续传能力**：✅ 支持

### 托管子任务机制

**"优雅取消"实现**：
1. 父 Task Flow 向子任务发送 SIGTERM 信号
2. 子任务捕获信号，保存当前进度
3. 释放文件句柄、网络连接
4. 向父 Task Flow 发送"已取消"确认
5. 父任务决策：重试/跳过/终止

### 对交易团队的价值

**适合场景**：
- 订单执行链：下单 → 确认 → 风控检查 → 成交记录
- 策略回测：数据拉取 → 指标计算 → 信号生成 → 报告生成
- 多币种同时监控：并行执行，汇总结果

**不适合场景**：
- 简单单轮查询
- 实时性极高的抢单操作（Task Flow 有状态序列化的延迟）
