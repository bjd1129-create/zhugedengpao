## 一、BetterClaw 七大稳定法则（深度补充）

**原文来源：betterclaw.io/blog/openclaw-best-practices**

### 新增细节
- **maxIterations 参数**：限制单次响应中连续工具调用次数，稳定设置 10-15（无此参数模型可链50+调用）
- **心跳模型降级**：Haiku ($1/$5 per M tokens) 或 DeepSeek ($0.28/$0.42 per M tokens) 用于每日48次心跳检查，节省70-80%成本
- **Spending cap 50%/80%双警报**：低于50%通知，低于80%警告，给调查留时间
- **Gateway 绑定 127.0.0.1 而非 0.0.0.0**：最常见配置错误，直接暴露API到互联网
- **ClawHub 恶意技能比例**：824+恶意技能，约20% registry，需安装前审查源码

### 第7条新增：版本控制配置
- openclaw.json、SOUL.md、自定义 skills 全部进 Git
- 每次变更 = commit，每个工作状态 = tag
- 更新后立即检查日志24小时（大多数问题在此窗口暴露）

## 二、Garrett's Field-Tested Best Practices v2（超详细架构指南）

**原文来源：github.com/garrettekinsman/openclaw-best-practices**

### 技能架构黄金法则
- **SKILL.md < 50行**：每多一行 = 每次触发吃掉上下文窗口 tokens
- **脚本 > 内联代码**：100行Python脚本 = 0 tokens（运行前不读）
- **scripts/ 放可执行逻辑，references/ 放文档，assets/ 放模板**
- **脚本设计原则**：CLI参数 + JSON结构化输出 + 最小化stdout + 内部错误处理

### 内存层次架构
```
MEMORY.md          → 精选长期记忆（精华，决定）
memory/YYYY-MM-DD  → 每日原始日志（发生了什么）
memory/archives/   → 已被提炼到MEMORY.md的旧上下文
HEARTBEAT.md       → 极简（每30-60分钟运行，每token都是钱）
```

### ContextGraph（新发现）
- 替代线性滑动窗口，改为基于标签的语义检索
- 只拉取与当前对话主题相关的消息，而非暴力加载最近历史
- ⚠️ 陷阱：MEMORY.md静态注入 vs ContextGraph动态检索——静态注入因出现更早而获胜，MEMORY.md过时内容会覆盖正确图谱检索结果

### 本地模型编排模式（重要）
```
Cloud模型（Opus/Claude）
    │
    ├── 聚焦任务prompt ──────────►│ 本地GPU模型
    │◄── 结构化响应 ─────────────┤
    ├── 验证数字（web search）    │
    ├── 错误审查                  │
    └── 最终综合                  │
```
**规则**：
- 本地模型（Qwen/LLaMA等）擅长生成，不擅长多步工具使用
- 不能导航文件系统——把所有相关代码直接粘贴到prompt
- 本地模型输出 = 不受信任：必须验证编译/逻辑/硬编码凭据/语言泄漏

### Ollama 关键参数
- `OLLAMA_KEEP_ALIVE=5m`，**永远不要设 -1**（共享GPU上无限保活会阻塞其他模型）
- `OLLAMA_MAX_LOADED_MODELS=1`（大模型超VRAM一半时必须）
- Tailscale IP 用于 provider config，而非 LAN IP（某些HTTP客户端对RFC 1918地址超时）

### 集群架构（已有架构对比）
```
[Gateway Mac Mini] ←→ [Compute Node 1 Framework/GPU] ←→ [Compute Node 2 Mac Studio]
  OpenClaw            Ollama + LiteLLM                  Ollama + MLX
  Tailscale           Tailscale                          Tailscale
```
- Heavy MoE (80B+) → 最高RAM节点
- Fast dense (7B-32B) → 最高带宽节点（Mac Studio）
- 计算节点不暴露公网端口

### Graceful Restart（安全关键）
```bash
# ✅ 正确：保持session存活
gateway config.patch（触发SIGUSR1）

# ❌ 错误：杀死连接，orphan sessions
openclaw gateway stop/restart
launchctl kickstart -k ...
kill -9 <gateway-pid>
```

## 四、Self-Improving Agent 四阶段闭环（完整架构）

**原文来源：blog.exit.st/openclaw-self-improving-agent-complete-guide/**

### Phase 1: Detection（检测）
监控自身性能，寻找能力缺口信号：
- 任务失败：无法完成或完成质量差
- 重复模式：同类请求反复出现但无专门处理程序
- 用户反馈：明确纠正或不满表达
- 效率指标：耗时过长或步骤过多

### Phase 2: Analysis（分析）
缺口检测后，分析所需：
- 缺少什么能力？
- 解决方案长什么样？
- ClawHub上已有相关技能？
- 修改现有技能还是全新创建？

### Phase 3: Generation（生成）
创建或修改技能：
- 编写新的 SKILL.md（含触发条件）
- 创建 scripts/ 支撑脚本
- templates/ 模板
- references/ 参考材料
- 测试技能（针对触发改进的原始场景）

### Phase 4: Integration（集成）
新/改进技能加入可用集：
- 技能文件放入正确目录
- 简短验证运行确认工作
- 改进日志供人类审查
- Agent立即开始使用新能力

### ClawHub 优先流程
```
检测能力缺口
    │
    ▼
搜索 ClawHub 现有技能
    │
    ├── 找到 → 安装并适配
    │
    └── 没找到 → 从头创建
              │
              ▼
        可选：发布回 ClawHub
```

### 版本控制
- 修改现有技能前保留前版本备份
- 记录：改了什么、为什么改
- 启用回滚（改进导致退步时可恢复）

### 衡量成功
| 指标 | 应有趋势 |
|------|---------|
| 任务成功率 | ↑ 上升 |
| 解决时间 | ↓ 下降 |
| 升级率 | ↓ 下降 |
| 创建的技能数 | ↑ 上升 |
| 改进速度（检测→部署） | ↑ 加快 |

## 五、Skill进化安全护栏（重要新增）

### 四类必须有人类审核
1. **Scope limits**：Agent只能在自己定义的类别内创建技能
2. **Resource limits**：技能生成不应消耗过多算力或API调用
3. **Review tiering**：低风险改进自动批准，外部集成必须人类审核
4. **Rollback capability**：每项改进必须可逆

### "Agent研究自我改进"场景的特殊风险
> Agent搜索自我改进技术时，会接触到那些来源中的任何内容。
> 恶意GitHub README或精心制作的博客文章理论上可能包含
> 旨在让Agent扩大自身权限的指令。
> SOUL.md中的明确禁止是一层防御。

**解决方案**：
- 永远不提出"给我更多权限"的变更
- 变更只涉及能力扩展，不涉及权限扩大
