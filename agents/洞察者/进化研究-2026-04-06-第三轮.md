## 一、OpenClaw-RL 论文深度（核心新发现）

### 1.1 核心洞察：Next-State Signals Are Universal

论文(arxiv:2603.10165, 2026/3/10发布，HuggingFace日榜第一)提出一个关键观察：

> 每次Agent交互后产生的"下一状态信号"——用户回复、工具输出、终端/GUI状态变化——本身就是通用训练数据，不需要额外标注。

**三种训练方法对比**：

| 方法 | 原理 | 信号类型 | 适用场景 |
|------|------|----------|----------|
| Binary RL | GRPO + PRM评分 | 标量奖励 | 个人助手优化 |
| OPD (On-Policy Distillation) | Hindsight-Guided蒸馏 | Token级方向优势 | 行为细节纠正 |
| Combined | Binary RL + OPD联合 | 标量+Token双信号 | 最强效果 |

### 1.2 OPD核心机制（最值得借鉴）

**Hindsight-Guided On-Policy Distillation**的工作流程：

1. 当Agent行为导致不好的下一状态时，Judge模型从中提取"本应怎么做"的文本提示
2. 用这个增强提示构建"更好的老师"
3. 学生(Token级logprob)与老师的差距成为比标量奖励更丰富的方向信号

**启示**：小花团队可以用类似思路做"对话复盘"——不是记录失败本身，而是记录"本应该怎么回复"。

### 1.3 重大更新timeline（来自GitHub）

- **2026/4/4**：支持基于多人反馈同时优化单一模型（Group Feedback Optimization）
- **2026/3/25**：Tinker云端支持
- **2026/3/20**：支持自有OpenClaw实例接入，只需安装extension
- **2026/3/13**：LoRA训练支持（此前只有全量微调）
- **2026/3/12**：支持LoRA训练
- **2026/3/10**：技术报告发布，HuggingFace日榜第一

### 1.4 异步四环架构（工程价值高）

```
Agent服务 ←→ Rollout收集 ←→ PRM/Judge评分 ←→ 策略训练
（独立运行，互不阻塞）
```

**关键**：模型在服务请求的同时，后台训练在运行，评判在并发打分。零协调开销。

### 1.5 对小花团队的适用性评估

**短期不适用原因**：
- 需要8×GPU硬件
- 需要CUDA 12.9 + Python 3.12环境
- 全量训练需要大量对话数据

**值得借鉴的思想**：
- OPD的" hindsight文本提示"思路 → 可迁移到每日复盘
- 异步并行的设计理念 → cron任务可参考
- 多人反馈优化（Group Feedback）→ 未来考虑老庄+女儿的反馈汇总

## 二、Self-Evolve.club 社区架构详解

### 2.1 核心机制三维价值

**强化稳定性**：用强化风格更新（而非记忆快照），行为更可靠
**RAG Token效率**：只检索相关经验三元组，降低上下文膨胀
**共享技能网络**：本地+远程记忆联合排名，贡献者直接提升全网效果

### 2.2 经验存储格式：Intent-Experience三元组

所有学习到的经验以结构化三元组存储：
```
Intent（意图） + Experience（经验） + Outcome（结果）
```

相比随意笔记，三元组更利于RAG检索。

### 2.3 Evolution Score计算公式

```
Evolution Score = Reuse Hits + Quality Reward
```

- **Reuse Hits**：记忆被其他节点复用的次数
- **Quality Reward**：高质量贡献的额外奖励

### 2.4 加入方式

```bash
npx clawhub@latest install self-evolve-skill
```

开启远程共享后，自动加入贡献者排行榜。

### 2.5 对小花团队的启示

**立即可用**：小花团队已安装self-evolve相关技能。可以：
- 查询self-evolve.club排行榜，了解全网进化水平
- 贡献经验到共享网络，换取Quality Reward
- 复用其他节点的优质经验

## 三、OpenClaw 2026最佳实践补充

### 3.1 模型选择原则（新原则）

**不要对简单任务用高端模型**：

```json
// 简单问答
{ "model": "gpt-5-3-codex" }

// 复杂推理
{ "model": "claude-4-5-sonnet" }
```

**小花团队的当前问题**：所有任务默认使用MiniMax-M2.7，没有按复杂度分流。这是可以优化的点。

### 3.2 缓存策略

```json
{
  "caching": {
    "enabled": true,
    "ttl": 3600,
    "maxSize": "100MB"
  }
}
```

OpenClaw Gateway支持内置缓存，可减少重复API调用。

### 3.3 成本管理配置

```json
{
  "limits": {
    "messagesPerMinute": 10,
    "messagesPerHour": 500,
    "maxTokensPerDay": 100000,
    "monthlyBudget": 50
  }
}
```

**小花团队目前没有设置成本上限**，建议评估。

### 3.4 健康检查与自动恢复

```json
{
  "monitoring": {
    "healthCheck": { "enabled": true, "interval": 60 }
  },
  "recovery": {
    "autoRestart": true,
    "maxRetries": 3,
    "retryDelay": 5000
  }
}
```
