## 一、Dreaming系统 — 最重要的新发现

### 1.1 三阶段分工（最核心的新知识）

Dreaming是OpenClaw内置的后台记忆整合系统，运行在memory-core插件中。三阶段各有明确分工：

| 阶段 | 功能 | 写哪里 | 类比 |
|------|------|--------|------|
| **Light** | 排序、去重( Jaccard)、聚类候选记忆 | DREAMS.md（仅inline模式） | 整理桌面 |
| **Deep** | 淘汰+打分 promotion到长期记忆 | **MEMORY.md**（唯一） | 归档文件 |
| **REM** | 模式识别、反思、高阶笔记 | DREAMS.md（inline模式） | 写日记 |

**关键边界**：Deep是唯一允许写MEMORY.md的阶段。Light和REM只操作临时文件，不直接修改长期记忆。

### 1.2 Deep阶段的评分机制

```json
{
  "deep": {
    "limit": 10,               // 每轮最多promote 10条
    "minScore": 0.8,           // 最低加权分数
    "minRecallCount": 3,       // 最少被recall 3次
    "minUniqueQueries": 3,    // 最少3个不同查询触发
    "recencyHalfLifeDays": 14, // 14天衰减一半
    "maxAgeDays": 30           // 超过30天的日志不promote
  }
}
```

**六信号评分**：recall次数、unique queries数量、最近一次recall时间、原始记忆年龄、内容质量权重、跨标签多样性。

### 1.3 Deep Recovery机制（自动健康恢复）

当长期记忆健康分数<0.35时，自动触发Recovery：
- 扫描过去30天日志找候选记忆
- 置信度≥0.97自动写入MEMORY.md
- 置信度0.9-0.97之间需要人工确认

这意味着**小花团队不需要手动监控记忆健康**，系统会自动补货。

### 1.4 对小花团队的适用性

✅ **已部分实现**：小花团队已有daily log + MEMORY.md机制，Dreaming是自动化版本
✅ **建议开启**：当前老庄的workspace可能没开Dreaming，建议确认memory-core插件状态
⚠️ **风险**：REM阶段产生的DREAMS.md如果长期不清理，可能造成额外token消耗

## 二、Session Pruning — 比Compaction更轻量的武器

### 2.1 Pruning vs Compaction核心区别

| | Pruning | Compaction |
|--|---------|------------|
| **操作对象** | 仅tool results（exec结果等） | 整个对话历史 |
| **是否落盘** | 否（内存级，per-request） | 是（写入session transcript） |
| **触发时机** | cache TTL过期后 | context window快满时 |
| **对话内容** | 完全不碰 | summarization |

### 2.2 核心价值：对Anthropic缓存友好

Pruning对Anthropic用户特别有价值：
- cache TTL过期后，完整prompt重新cache
- Pruning减少cache写入大小 → 直接降低成本
- Anthropic API key用户默认开启（30min heartbeat）

### 2.3 对小花团队的建议

当前配置用MiniMax-M2.7，**建议手动开启Pruning**：
```json
{
  "agents": {
    "defaults": {
      "contextPruning": { "mode": "cache-ttl", "ttl": "5m" }
    }
  }
}
```
MiniMax的context window比Claude小很多，Pruning可以延缓compaction触发，保留更多近期对话上下文。

## 三、Standing Orders — 自主执行授权体系

### 3.1 Execute-Verify-Report循环（核心理念）

Standing Orders的核心不是"告诉代理做什么"，而是"授权代理自主做决策"。每个程序遵循：

1. **Execute** — 做实际工作（不是只回复"好的"）
2. **Verify** — 确认结果正确（文件存在？消息发出？数据解析成功？）
3. **Report** — 告知owner结果+异常

### 3.2 对小花团队的直接应用

**当前问题**：小花每天的heartbeat只是"检查"，没有形成"执行-验证-报告"闭环。

**建议增加**：
```
## Program: Daily Evolution Review

Authority: 检查performance-log.json、读取昨日日志、评估记忆健康
Trigger: 每日 Heartbeat（8AM）
Approval gate: 发现重大异常才上报，正常情况自主处理
Escalation: 日志中发现同类型错误3次以上 → 报告老庄

Steps:
1. 读取 memory/YYYY-MM-DD.md 和 performance-log.json
2. 评估：今日任务完成度、错误模式、token消耗
3. 如果发现问题 → 写入 MEMORY.md 警示条款
4. 如果无异常 → 静默，不打扰老庄
```

## 四、Hooks体系 — 事件驱动的自我感知

### 4.1 最有价值的Hook事件

| 事件 | 触发时机 | 进化价值 |
|------|----------|----------|
| `session:compact:before` | 压缩前 | 自动保存重要上下文 |
| `session:compact:after` | 压缩后 | 验证压缩质量 |
| `agent:bootstrap` | 加载workspace文件前 | 动态修改启动上下文 |
| `gateway:startup` | 网关启动 | 初始化自检 |
| `message:preprocessed` | 消息预处理后 | 情绪检测、意图分类 |

### 4.2 小花团队可用的Bundled Hooks

| Hook | 功能 |
|------|------|
| `session-memory` | 每次/new或/reset自动保存session到memory/ |
| `bootstrap-extra-files` | 从glob pattern注入额外bootstrap文件 |
| `command-logger` | 记录所有命令到~/.openclaw/logs/ |
| `boot-md` | Gateway启动时运行BOOT.md |

**建议开启** `session-memory` + `command-logger`，用于追踪自我改进轨迹。
