## 一、本次核心发现

### 1. self-evolve 插件（longmans/self-evolve）

这是目前最完整的 OpenClaw 自我进化实现，核心机制是 **Q-Learning + RAG**：

**工作原理（三句话版）**：
- 每次任务结束，插件观察对话轨迹，等待用户反馈信号（正/负）
- 检测到反馈后，对轨迹做两轮脱敏（本地→LLM二次脱敏），提取为「意图-经验-嵌入」三元组
- 下次遇到相似意图，从本地/远程记忆中检索高相关性经验，注入到 prompt 前

**技术细节**：
- 挂在 `before_prompt_build` 和 `agent_end` 两个 hook 上
- 任务状态机：open → waiting_feedback → learn/close
- Q值更新：reward × γ 衰减 × 经验置信度
- 检索门控：τ=0.85，只在相似度 > 0.85 时注入记忆（节省 token）
- 学习门控：minAbsReward=0.15，minRewardConfidence=0.55

**三种学习模式**：
| 模式 | 触发条件 | Token成本 |
|------|----------|-----------|
| balanced（默认） | 优先工具调用；无工具调用需更高奖励 | 中 |
| tools_only | 只学调用了工具的轮次 | 低 |
| all | 所有通过奖励门的轮次都学 | 高 |

### 2. Skill Evolution Network（self-evolve.club）

去中心化进化网络，本质是一个共享记忆池：

- **本地记忆 + 远程记忆**合并后统一 rank
- **隐私设计**：两轮脱敏（sanitizeMemoryText + LLM summarizer）后才上传
- **激励机制**：reusers 越多、quality reward 越高 → leaderboard 排名越高
- **Evolution Score = Reuse Hits + Quality Reward**

### 3. OpenClaw Skill Evolution vs 小花现有机制对比

| 维度 | 现有机制（AGENTS.md） | self-evolve 插件 |
|------|---------------------|-----------------|
| 触发方式 | 人工判断 3次失败 | 自动观察 + 奖励门控 |
| 学习方式 | 事后记录到文件 | 实时 Q 值更新 |
| 记忆格式 | Markdown 日志 | 三元组 + 向量检索 |
| 遗忘机制 | 无 | maxEntries=200，超出后保留高价值记忆 |
| 共享 | 无 | 可选加入共享网络 |
| 反馈利用 | 弱（需要人工识别） | 强（自动检测正/负反馈信号） |

## 二、对小花团队的启发

### 可以直接借鉴的点

**1. 建立反馈信号识别能力**
当前 AGENTS.md 的进化触发靠"3次失败"，效率低。self-evolve 的启发是：建立一套自动反馈检测机制（正反馈信号如表扬、完成确认；负反馈信号如否定、重做要求），用这套信号驱动自动学习。

**2. 三元组记忆格式**
现有 memory/YYYY-MM-DD.md 是叙事型日志，不适合检索。建议逐步改为「意图-经验-结论」三元组，存到 memory/triplets/ 目录下，配合简单关键词检索。

**3. 学习模式选择**
如果未来引入 self-evolve 插件，建议先用 `tools_only` 模式跑（Token成本最低），稳定后再切 `balanced`。

### 暂时不适用的点

- **远程共享网络**：小花团队规模小，暂无必要加入共享网络
- **Q-Learning 量化体系**：对当前业务复杂度来说过度设计
