# 技术架构培训
## 复杂系统设计

**前言：架构是取舍的艺术**

技术架构不是画图，不是选最新的框架，不是追最炫的技术。架构是关于「在特定约束条件下，如何做出能支撑业务长期发展的技术决策」。

一个只有 1000 用户的产品和一个有 1000 万用户的产品，架构可能完全相同，也可能天壤之别——关键在于你们现在处于什么阶段，面临什么约束，未来会走向哪里。

本培训不是为了让你成为架构师，而是让你理解：为什么系统要这样设计，以及当你参与架构讨论时，如何提出有价值的问题。

---

## 一、架构的本质：分与合的艺术

### 1.1 什么是架构

简单说：**架构是对系统进行分块，以及规定块与块之间关系的方式。**

分得太散 → 系统复杂、协作成本高、数据一致性难
分得太粗 → 扩展性差、一个模块挂了全挂、团队耦合

架构的核心问题只有一个：**怎么分，才能让系统跑得稳、改得快、成本低？**

### 1.2 架构不是设计出来的，是演化出来的

很多团队一开始就想要一个「完美架构」，画了十几层图，用了最先进的技术，结果产品没跑起来，技术债倒背了一身。

**更聪明的做法：**
- 先用最简单的方案跑起来
- 等业务验证了，再针对性扩展
- 始终让架构匹配当前的业务复杂度

> 不要为 1000 万用户设计架构，除非你已经服务了 100 万用户。

### 1.3 架构决策的三个维度

每个架构决策，本质上都在权衡三个东西：

| 维度 | 考虑的问题 |
|-----|----------|
| **功能性** | 这个设计能解决业务问题吗？ |
| **可扩展性** | 业务增长 10 倍，系统还能撑住吗？ |
| **成本** | 开发、维护、运行的成本能接受吗？ |

好的架构决策，是在这三者之间找到当前最优解。

## 二、核心架构原则

### 2.1 高内聚、低耦合

这是软件架构最经典的原则，展开说：

**高内聚**：每个模块只做好一件事，相关的代码都放在一起。
- 好处：理解成本低，修改影响范围小，测试容易

**低耦合**：模块和模块之间的依赖要尽可能少，接口要稳定。
- 好处：一个模块出问题不影响其他模块，可以独立替换

**实操检验**：如果你要改一个功能，需要同时改 5 个模块的代码，那说明你的模块边界设计有问题。

### 2.2 面向失败设计

好的架构师，花在「系统挂了怎么办」上的心思，比花在「系统怎么跑」上的心思还多。

**常见故障场景：**
- 数据库挂了 → 应用还能读缓存、降级显示
- 某台服务器挂了 → 流量自动切换到其他服务器
- 第三方 API 超时 → 本地做兜底，不卡死主流程

**核心思维：**
- 任何依赖都可能是不可靠的
- 任何服务都可能是会挂的
- 任何数据都可能是会丢的

基于这个假设去设计，才能真正提高系统的健壮性。

### 2.3 不要重复造轮子

这句话有两层意思：

**第一层**：业界已有的成熟方案，能用就用。
- 数据库用 PostgreSQL/MySQL，不要自己写存储引擎
- 缓存用 Redis，不要自己写内存缓存
- 消息队列用 Kafka/RabbitMQ，不要自己写发布订阅

**第二层**：你们团队内部已经有的能力，不要重新开发。
- 先调研现有模块能不能复用
- 先看看其他项目有没有类似的实现

## 三、常见架构模式

### 3.1 单体架构

**定义**：整个应用打包成一个部署单元，所有代码在一个项目里。

**优点**：
- 开发、部署、测试都简单
- 早期项目启动快
- 调试方便，本地跑起来就能测

**缺点**：
- 代码量大之后编译慢、启动慢
- 一个模块的 bug 可能拖垮整个系统
- 技术栈一旦选了就难换

**适用场景**：用户量 < 1 万，团队 < 10 人，功能还在快速探索期。

### 3.2 微服务架构

**定义**：把系统拆成多个独立部署、独立运行的小服务，每个服务做一件事。

**优点**：
- 每个服务可以独立开发、部署、扩展
- 技术选型灵活，不同服务可以用不同语言
- 故障隔离，一个服务挂了不影响其他

**缺点**：
- 分布式系统的复杂度陡然上升
- 服务间通信、数据一致性、服务治理都是挑战
- 本地开发环境搭建复杂

**适用场景**：用户量 > 100 万，团队 > 50 人，有足够的技术储备。

### 3.3 前后端分离

**定义**：前端和后端完全分开，前端通过 API 和后端交互。

**优点**：
- 前端可以单独部署、单独迭代
- 后端 API 可以同时支持 Web、App、小程序
- 团队职责清晰

**缺点**：
- SEO 困难（需要 SSR 或预渲染方案）
- 首屏加载性能需要额外关注
- 前后端联调增加了协作成本

**适用场景**：几乎所有现代 Web 应用。

### 3.4 事件驱动架构

**定义**：系统通过事件（消息）来驱动业务流程，而不是直接调用。

**举例**：用户下单 → 发布「订单创建」事件 → 库存服务订阅并扣库存 → 物流服务订阅并发货

**优点**：
- 系统之间天然解耦
- 容易扩展新功能（新服务订阅事件即可）
- 峰值流量可以被消息队列缓冲

**缺点**：
- 事务一致性处理复杂（需要引入 Saga、最终一致性等模式）
- 调试和追踪困难（需要完整的分布式追踪工具）
- 业务链路不直观

## 四、数据架构基础

### 4.1 数据库选型

**关系型数据库（MySQL/PostgreSQL）**：
- 数据结构清晰，有表之间的关联关系
- 事务支持好，数据一致性有保障
- 适合核心业务数据、用户数据、订单数据

**NoSQL 数据库**：
- MongoDB：文档型，适合数据结构经常变化的场景
- Redis：KV 型，适合缓存、临时数据、会话管理
- Elasticsearch：搜索型，适合全文检索、日志分析

**选型原则**：
- 90% 的场景 MySQL 够了
- 不要为了「潮」选不熟悉的数据库
- 混用是允许的，但要有明确的理由

### 4.2 缓存策略

**为什么要缓存**：数据库读写是系统最常见的瓶颈之一。缓存可以让你用内存换速度。

**缓存的层次**：
1. **本地缓存**：进程内的 HashMap、LRU Cache。快，但不可共享
2. **分布式缓存**：Redis。跨进程共享，是最常用的方案
3. **CDN 缓存**：静态资源（图片、JS、CSS）。离用户近，效果最好

**缓存三问**：
1. 缓存的数据会不会过时？（缓存过期策略）
2. 缓存和数据库的数据怎么保持一致？（缓存更新策略）
3. 缓存挂了怎么办？（降级策略）

### 4.3 数据库读写分离

**原理**：一个主库（写），两个从库（读）。写操作走主库，读操作走从库。

**适用场景**：读多写少的业务（大多数 ToC 产品都是）。

**注意**：主从同步有延迟，读从库可能读到稍旧的数据。如果业务对「读到最新数据」要求高（比如金融场景），这个方案不适用。

## 五、高可用与容错

### 5.1 负载均衡

**原理**：用户的请求不是打给某一台服务器，而是先打到负载均衡器，由它分配到具体的服务器。

**常见方案**：
- Nginx（最常用，配置简单）
- 云服务自带 ALB/CLB（阿里云、AWS 都有）

**负载均衡策略**：
- 轮询：每个请求依次分配
- 最少连接：分配给当前连接数最少的服务器
- IP Hash：同一个 IP 的请求固定打到某一台服务器（适合有状态的场景）

### 5.2 服务冗余

核心思想：任何一个服务，都至少有两台实例在跑。任何一台挂了，系统还能正常服务。

**实操**：
- 应用服务器：至少 2 台，用负载均衡
- 数据库：主从复制，主库挂了从库顶上
- Redis：主从 +哨兵，或者 Cluster 模式

### 5.3 熔断与降级

**熔断**：当某个服务的调用失败率超过阈值时，自动「熔断」，不再调用这个服务，直接返回降级数据。

**降级**：当系统压力过大或某个功能不可用时，主动关闭非核心功能，保证核心功能正常。

**类比**：就像你家电路跳闸了（熔断），你不修电路，而是先点蜡烛（降级）。

## 六、架构评审 checklist

当你参与一个架构评审，或者自己要设计方案时，用这个 checklist 检查：

### 需求层面
- [ ] 真的理解业务需求了吗？有找业务方确认过吗？
- [ ] 性能要求是什么？QPS、延迟、可用性，有量化指标吗？
- [ ] 未来的扩展方向是什么？这个设计能撑多久？

### 技术层面
- [ ] 每个组件的职责边界清晰吗？
- [ ] 数据流向是什么？从请求到响应，数据走了哪些地方？
- [ ] 如果某个组件挂了，系统还能工作吗？哪些功能会受损？
- [ ] 未来的技术债是什么？现在做的妥协，未来怎么偿还？

### 运维层面
- [ ] 新机器准备好了吗？扩容需要多长时间？
- [ ] 监控报警做了吗？挂了能第一时间知道吗？
- [ ] 数据有备份吗？能恢复到哪个时间点？
- [ ] 日志完整吗？能追踪一次请求的完整链路吗？

## 七、给非技术人员的架构建议

如果你不是技术背景，参与技术讨论时记住三点：

1. **问场景，不问技术**：问「这个方案能解决我们现在的什么问题」，而不是「这个技术是不是最先进的」

2. **问代价，不问原理**：问「换这个技术需要多少时间，会停服多久」，而不是「这个技术底层是怎么实现的」

3. **问长期，不问短期**：问「这个方案一年后还适用吗」，而不是「现在能不能快点上线」

---

## 总结

架构的核心逻辑很简单：**在当前约束条件下，用最小的成本，让系统可靠地支撑业务。**

记住：
- 架构是演化的，不是设计出来的
- 最简单的方案，往往是最好的起点
- 高可用不是目的，业务可持续才是目的
- 每个架构决策都有代价，要清楚地知道你在为什么付出

**好的架构，让业务跑得更快；好的架构师，让团队不需要意识到架构的存在。**
