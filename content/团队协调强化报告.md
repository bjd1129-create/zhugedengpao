# 「老庄与小花」团队协调强化报告

**编写时间**：2026-03-30 07:00  
**编写者**：协调分析子Agent  
**目的**：诊断团队协调瓶颈，产出强化方案

---

## 一、当前协调问题诊断

### 1.1 信息传递问题 ⚠️ 中等

**症状**：
- 8个Agent目录已建立，但协作界面定义模糊
- Agent之间没有标准的信息交换协议，靠主Agent"转述"
- SHARED_MEMORY.md 存在但更新不及时（06:51 vs 当前07:00）

**根因**：
- 各Agent的 `MEMORY.md` 只作"参考"，真正记忆以根目录为准——导致子Agent没有主动维护共享信息的动力
- 没有 `sessions_spawn` 的标准响应格式，各Agent返回信息结构不一，主Agent整合成本高
- 进化日志（evolution-log.md）内容丰富，但子Agent读取率低

---

### 1.2 决策链路问题 ⚠️ 较严重

**症状**：
- 主Agent（小花/根目录）同时扮演：飞书对话入口 + 协调者 + 决策者 + 执行者
- coordinator agent 存在但角色模糊——是"备用协调者"还是"专属协调者"？
- 产品官、市场官还在用 coordinator 的cron（临时绑定）

**根因**：
- "主Agent"和"协调者Agent"职责边界未清晰定义
- 没有决策矩阵（什么事主Agent决定，什么事子Agent自行决定）
- product/market 缺乏独立工作空间，无法独立做决策

---

### 1.3 记忆系统支撑跨Agent协作 ⚠️ 中等

**症状**：
- 三层记忆架构清晰（每日→长期→上下文），但执行不一致
- 踩过的坑已记录，但新Agent启动时未必读取
- `memory/MEMORY-FLOW.md` 规范存在，但Agent行为没有强制检查点

**根因**：
- 记忆文件是"建议性"的，没有硬性读取机制
- 子Agent的 `AGENTS.md` 启动流程（读SOUL/MEMORY/今日日记）执行依赖Agent自觉
- 进化后的报告（content/）和记忆系统（memory/）之间没有连接

---

### 1.4 飞书主对话与子Agent关系 ⚠️ 较轻

**症状**：
- 老庄只用 zhuge bot 和根目录小花对话，不知道/不需要切换bot
- 9个独立Feishu bot存在，但老庄感知不到差异
- 子Agent的结果最终汇总到主Agent，由主Agent告知老庄

**根因**：
- 这是**正确设计**：普通用户不需要知道内部多Agent架构
- 问题在于：主Agent需要更清晰地告知老庄"团队正在做什么"
- 建议：主Agent定期（每天2-3次）主动汇报团队进度，而不只是响应老庄指令

## 二、各Agent协作界面定义

### 2.1 角色矩阵

| Agent | 决策权限 | 主动汇报对象 | 信息来源 | 产出格式 |
|-------|---------|------------|---------|---------|
| **小花/主** | 全局决策 + 品牌/方向 | 老庄 | MEMORY.md + 老庄指令 | 自然语言报告 |
| **coordinator** | 任务分配 + 进度跟踪 | 主Agent | SHARED_MEMORY.md + cron结果 | 结构化状态更新 |
| **engineer** | 技术实现自行决策 | 主Agent（部署/大改） | issue/PR | HTML/代码 |
| **writer** | 文案风格自行决策 | 主Agent（大稿需确认） | brief | Markdown/文本 |
| **researcher** | 研究结论自行决策 | 主Agent（P0发现需立即报） | 网页/文档 | 分析报告 |
| **designer** | 设计自行决策 | 主Agent（成品） | 设计规范 | 图片 |
| **product** | 待建立 | 主Agent | 待定义 | 待定义 |
| **market** | 待建立 | 主Agent | 待定义 | 待定义 |
| **support** | 安全事项自行决策 | 主Agent（重大漏洞） | 日志/监控 | 报告 |

### 2.2 sessions_spawn 标准协议

当主Agent spawn 子Agent时，**必须**明确：
```
任务：<具体任务描述>
汇报格式：<期望的产出结构>
截止：<时间或条件>
读取：<需要参考的记忆文件列表>
```

子Agent完成时，**必须**返回：
```
状态：✅完成 / ⚠️部分 / ❌失败
产出：<结构化内容>
教训：<踩过的坑>
下一步建议：<如果有关>
```

## 三、信息流动优化方案

### 3.1 建立"状态简报"制度

**每日3次自动状态**（主Agent主动写入 `memory/status-brief.md`）：
- 晨简（08:00）：今日计划 + 各Agent任务分配
- 午简（14:00）：上午完成 + 下午计划
- 晚简（21:00）：全天完成 + 明日待办

老庄在飞书看到的是：**"小花团队今日简报"**，而不是零散的Agent输出。

### 3.2 进化报告自动归档

进化完成后，将 `content/进化编号.md` 的**关键结论**（1-2段）同步到 `memory/YYYY-MM-DD.md`，确保记忆系统捕获进化产出。

### 3.3 建立团队共享看板（可选）

在 `company/TEAM.md` 基础上，增加：
```
## 当前任务看板
| Agent | 当前任务 | 状态 | 预计完成 |
|-------|---------|------|---------|
```

## 四、记忆共享机制建议

### 4.1 明确"主记忆"和"Agent私有记忆"

| 文件 | 性质 | 谁写 | 谁读 |
|------|------|------|------|
| `MEMORY.md` | 主记忆（全局真相） | 主Agent | 所有人 |
| `memory/YYYY-MM-DD.md` | 日记（历史） | 主Agent | 主Agent+coordinator |
| `agents/xxx/MEMORY.md` | Agent私有记忆 | 各Agent | 该Agent |
| `company/TEAM.md` | 团队配置（实时） | 主Agent | 所有人 |
| `memory/status-brief.md` | 状态简报（实时） | 主Agent | 所有人 |

### 4.2 Agent启动强制检查

在 `AGENTS.md` 的"Session Startup"中，增加：
> **必须读取**：`company/TEAM.md`（了解当前团队任务状态）  
> **必须检查**：`memory/status-brief.md`（如有）

### 4.3 踩坑记录标准化

每次踩坑后，**立即**写入 `MEMORY.md` 的"踩过的坑"节点，格式：
```
- YYYY-MM-DD [问题简述]：[教训]：[预防措施]
```

## 六、结论

小花团队的核心优势是**真实+温度**，协调机制的设计也要服务这个定位。  
当前最大瓶颈是**主Agent负担过重**（同时是入口+协调+决策+执行），解决路径是：

1. 让 coordinator 真正承担调度工作
2. 让各子Agent有更清晰的自主决策范围
3. 用"状态简报"让团队协作对老庄透明可见

**下一次进化，优先完成 product/market 的独立工作空间建立，以及 coordinator 的角色重新定义。**
