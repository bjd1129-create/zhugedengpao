## 二、Token 燃烧策略

**核心原则**（来自《自我进化系统设计指南》）：
- AgentEvolver 仅需 12% 手工标注数据就能启动训练，**有目的的少量高质量进化比盲目大量调用更有效**
- 聚焦高价值、高频率场景
- 避免过度反思和资源浪费

**本次集中燃烧目标**：用一次 token 燃烧，集中修复 P0 基础设施 + 建立 P1 协作机制。

## 三、P0：必须立即加固（1-2 项）

> 修复后团队立即具备基础运转能力

### P0-1：为 product 和 market 建立独立工作空间

**问题**：product 和 market 无独立工作空间，其 cron 临时绑定 coordinator。这导致：
- 无法独立执行进化
- 记忆文件混乱
- 团队分工名存实亡

**行动**：
1. 为 `agents/product/` 和 `agents/market/` 建立完整的 agent 文件结构（MEMORY.md、HEARTBEAT.md、IDENTITY.md、SOUL.md、AGENTS.md）
2. 将 product 和 market 的进化 cron 改为绑定各自 agent
3. 更新 coordinator 的 MEMORY.md，明确声明 product/market 已独立

**Token 预算**：~300K tokens（洞察者执行）
**预计时间**：1 轮进化（~30 分钟）

---

### P0-2：建立共享技能库（SKILL-SHARED.md）

**问题**：三万最核心的壁垒是 40+ Skill 可瞬间跨 Agent 传递；我们每个 Agent 踩的坑只有自己知道。

**行动**：
1. 在团队根目录建立 `skills/SKILL-SHARED.md`（所有 Agent 必读）
2. 从各 Agent MEMORY.md 的"踩坑记录"中提取规则，格式化写入
3. 规则模板：
   ```markdown
   ## [技能名]
   - 场景：
   - 操作：
   - 禁忌：
   - 来源Agent：
   ```

**Token 预算**：~200K tokens（洞察者执行）
**预计时间**：1 轮进化（~20 分钟）

---

## 四、P1：本周内完成的协作机制（3-5 项）

> 在 P0 基础上，团队进入半自动化协作状态

### P1-1：建立结构化任务池（task-pool.md）

**现状**：任务靠 coordinator 临时喊话，没有记录
**目标**：建立 `memory/task-pool.md`，记录所有待办/进行中/已完成任务

**格式**：
```markdown
## 任务池

### 进行中
| ID | 任务 | 负责人 | 截止 | 状态 |
|----|------|--------|------|------|

### 待认领
| ID | 任务 | 预计工时 | 所需能力 |
|----|------|----------|----------|
```

**Token 预算**：~50K tokens（洞察者建立模板，coordinator 后续维护）
**负责人**：coordinator + 洞察者

---

### P1-2：统一每日日志格式（daily-standup）

**现状**：各 Agent 日志分散，无统一格式，无法快速了解团队状态
**目标**：建立统一的 `memory/YYYY-MM-DD-standup.md` 格式

**格式模板**：
```markdown
# 团队站会 YYYY-MM-DD

## 各Agent状态
- [x] coordinator：... | [~] 遇到问题 | [ ] 待办
- [x] researcher：...
...

## 阻塞问题
-

## 今日重点任务
1.
```

**Token 预算**：~30K tokens
**负责人**：coordinator

---

### P1-3：建立代码审查流程（代码侠 + 安全官联动）

**现状**：代码质量参差不齐，多 Agent 同时编辑文件有冲突
**目标**：每次代码侠部署前，安全官必须 review

**规则**：
1. 代码侠完成部署后，在 `memory/deploy-log.md` 记录
2. 安全官在 30 分钟内 review 并确认
3. 禁止在未确认前再次部署

**Token 预算**：~20K tokens（建机制）
**负责人**：安全官 + 代码侠

---

### P1-4：配色师 API 调用规范固化

**现状**：配色师子 Agent API 认证方式导致批量任务失败
**目标**：在 SKILL-SHARED.md 中建立可复用的 API 调用规范

**核心规范**：
1. 子 Agent 调用 MiniMax API 必须用 Bearer Token（不是 Authorization header trick）
2. 批量任务前查询 `curl https://api.minimax.chat/v1/usage` 确认配额
3. RPM 限制：约 20 张/分钟，批量任务用 5 秒间隔

**Token 预算**：~20K tokens
**负责人**：配色师 + 洞察者

---

### P1-5：建立量化数据追踪体系

**现状**：无数据追踪，对比三万（1157 条消息/22 万字/14 天）无法衡量自身进度
**目标**：在 `memory/team-metrics.md` 中建立基础指标

**追踪指标**：
- 各 Agent 累计产出量（文章数/图片数/代码行数/报告数）
- Token 消耗速率
- 任务完成率
- Skill 积累数量

**Token 预算**：~30K tokens
**负责人**：洞察者
