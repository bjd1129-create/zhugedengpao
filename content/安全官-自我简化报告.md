## 一、我是谁——安全官的双重职能

### 职能1：安全防护
- 代码安全审查、密钥规范、内容合规、API管理
- 制定安全红线并守护

### 职能2：Token消耗监控（新增）
- 总配额：4500次/5小时
- 每30分钟检测一次消耗
- 低于70%警戒线时触发提醒

**核心矛盾：两个职能叠加 = 安全官本身也是token消耗大户。**

## 二、流程问题——为什么安全官也在烧token

### 当前冗余

| 环节 | 问题 | 浪费点 |
|------|------|--------|
| 安全审查 | 每次心跳都做完整检查 | 重复扫描 |
| Token监控 | 每30分钟精确计算 | 大量小调用 |
| 提醒触发 | 每次都通知老庄 | 打扰过多 |
| 报告产出 | 追求全面安全报告 | token密集型 |

**核心问题：监控本身也在消耗大量token，形成负向循环。**

---

## 三、简化后的工作流

### Before（烧token版）
```
每30分钟：精确统计token → 计算消耗率 → 判断是否触发警戒 → 记录memory → 通知小花 → 小花通知老庄
```

### After（精简版）
```
每小时末：扫一眼进度 → 只在真正危急时报一次 → 其余静默
```

**三不原则：**
- 不做重复性全面扫描
- 不精确计算每次消耗（大概有数就行）
- 不频繁打扰老庄（除非真的要爆了）

## 四、最需要守住的2条安全红线

### 🔴 红线1：API密钥绝不能硬编码
**规则：** 代码中出现任何密钥字样（api_key、secret、token、password）立即报警  
**处理：** 立即阻断，通知老庄  
**原因：** 这是最常见的数据泄露源头，一次硬编码 = 永久暴露

### 🔴 红线2：用户数据绝不传出系统
**规则：** 任何涉及用户信息的内容，只能在系统内部流转  
**处理：** 发现即阻断，要求说明用途  
**原因：** 隐私合规底线，不可突破

## 五、Token监控简化流程

### 当前问题
每30分钟精确检测 → 大量小调用本身就在烧token

### 简化方案：心跳合并 + 静默优先

**具体做法：**
1. **心跳检查（每1小时）** → 顺手看一眼剩余token，大概估算够不够用
2. **不精确计算** → 只判断"明显够"还是"明显不够"，不求百分比
3. **只报一次** → 从"每次心跳都提醒"改为"整个周期只报一次"
4. **触发条件收紧** → 从70%警戒线收紧到只剩1格时才报
5. **提醒链路简化** → 直接写memory，不走小花二次通知

**简化后流程：**
```
每小时心跳 → 扫一眼 → 够用就静默 → 真的要爆了才写memory
```

**什么时候才通知老庄：**
- Token只剩最后1小时配额时
- 某个agent出现异常大量消耗时

**什么时候静默：**
- 进度正常时
- 消耗低于预期时（省token是好事）
- 静默期（22:00-08:00）

## 六、当前最需要检查的3个安全项

### 🚨 检查1：API密钥现状（最高优先）
**检查内容：** 整个工作区是否有硬编码的密钥、token、secret  
**风险：** 若已泄露，攻击者可调用团队资源  
**行动：** 快速grep搜索 `api_key`、`secret`、`token`、`password` 关键词

### 🚨 检查2：代理/VPN状态
**检查内容：** 飞书发国内社媒前是否忘记关代理  
**风险：** 发帖暴露真实IP，可能被平台限流或封号  
**行动：** 检查当前网络环境配置

### 🚨 检查3：外部命令执行记录
**检查内容：** 近期是否有未经老庄确认的外部命令执行  
**风险：** 未知来源的命令可能造成数据泄露或资源浪费  
**行动：** 回顾今日exec命令日志
