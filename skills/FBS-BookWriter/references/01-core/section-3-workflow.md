## §3.0.5 强制联网检索（写作质量门禁）

> **原则**：凡涉及事实、数据、政策、竞品、工具能力、行业结论的表述，必须先检索再写入；禁止凭模型记忆直接落稿。配置与阶段清单见 [`../05-ops/search-policy.json`](../05-ops/search-policy.json)。

| 阶段 | 强制动作 | 说明 |
|------|----------|------|
| **S0** | 竞品/读者/市场类查询均须执行检索 | **宿主**侧用 WebSearch/WebFetch 完成；`search-policy.json` 中 `s0ParallelQueries` 为建议轮次 |
| **S1** | 定调前检索主题与受众相关公开讨论/定义 | 用于校准「核心主张」与差异化 |
| **S2** | 目录设计前检索同类书目结构与空白点 | 至少 1 轮结构化检索 |
| **S2.5** | 脉络与**本书变现**路径须有检索支撑；**市场/行业**信息仅作交叉验证的背景论据 | 与 S0 数据交叉验证，避免用赛道叙事顶替本稿变现 |
| **S3 每章** | **每章成文前 ≥2 次**主题相关检索（默认） | 次数以 `search-policy.json` 的 `chapterWriting.minQueriesPerChapter` 为准；宿主应记录检索次数以便审计 |
| **S5** | 终稿事实与数据与「数据保鲜表」对齐前的抽查检索 | 对高风险句逐条验证 |

### 「禁止无检索落稿」单一禁令（与 SKILL / policy 对齐）

以下仅为**同一禁令**在不同文件中的互补表述，执行时**取严**，勿因措辞差异降低标准：

| 来源 | 作用 |
|------|------|
| [`SKILL.md`](../../SKILL.md) §0 **联网搜索规范** | 主规范叙事与用户可见承诺 |
| 本节 §3.0.5 上表 | 按阶段的操作化清单 |
| [`search-policy.json`](../05-ops/search-policy.json) | 机读阶段列表与每章最少检索次数 |

**禁令核心**：凡应核验的事实型/数据型/政策竞品型内容，**须先 WebSearch/WebFetch（或宿主等价工具）再落稿**；禁止仅用模型内省充当「已查证」。用户声明纯虚构并自担风险时，须在对话与产出物中**可见标注**。

**宿主**侧：用自然语言要求并行成员先输出「检索 query 列表 + 结果摘要」再动笔；本技能包**不含自研后端**，门禁由宿主按本规范落实。

## §3.0.55 阶段顺序与禁止跨阶段

> **目的**：避免未确认的定位/目录被后续成文「锁死」，或终稿流程倒序执行导致审计失效。

| 规则 | 说明 |
|------|------|
| **顺序** | 默认严格按 **S0（可跳过，见下）→ S1 → S2 → S2.5 → S3 → S4 → S5 → S6**；前一阶段用户确认或明确放行后，再进入下一阶段。 |
| **禁止跨阶段** | 不得在 **未交付/未确认** S1 定位稿的情况下进入 S2 目录定稿；不得在 **未确认** S2.5 脉络的情况下批量启动 S3 全书写入；不得跳过 S5 终审直接对外宣称「定稿发布」（除非用户书面确认接受预览版并自担风险）。 |
| **例外** | 用户明确要求「只改第 N 章」时，允许在已存在经确认的全书骨架下**局部回退 S3**，但仍须遵守该章的 **章前检索门禁**（`search-policy.json`）。 |
| **审计** | 宿主应记录「当前阶段 ID + 最近确认时间」，供质量报告与 `integration/CHECKLIST.md` 对照。 |

## §3.0 隐式确认最低确认清单

> **目的**：规范用户确认行为，避免"过少致失控感、过多致中断感"  
> **设计原则**：审批点设在"方向性决策"节点，非"执行细节"节点

### A层：自动确认（极低风险，无需确认）

| 场景 | 示例 | 处理方式 |
|------|------|----------|
| 错别字修正 | 的/地/得错误 | 自动修正 |
| 标点规范化 | 引号/省略号全角半角 | 自动规范化 |
| 格式统一 | 标题层级对齐模板 | 自动对齐 |
| 数据保鲜标注 | 来源URL补全 | 自动补全 |

### B层：隐性确认（必做，用户无感知打断）

| 节点 | 确认项 | 展示方式 |
|------|--------|----------|
| **S1-C 定位确认** | 主题 + 读者 + 风格 + 篇幅 + 大纲 | 定位确认卡（一屏展示5项） |
| **S2.5 脉络确认** | 内容方向 + 变现路径建议 | 脉络摘要卡 |
| **S3 大纲审批** | 章节结构 + 核心论点 | 结构预览图；**进入 S3 成文前须显性确认**（见阶段2「S2→S3 硬性门禁」） |
| **S5 终稿预览** | 全文质量报告 + 评分卡 | 评分详情卡 |

### C层：显性确认（必做，阻断执行）

| 场景 | 阻断条件 | 确认方式 |
|------|----------|----------|
| **删除章节** | 任何删除操作 | 弹窗确认，显示影响范围 |
| **方向变更** | S1→S2后改主题 | 显性确认 + 影响说明 |
| **定稿发布** | 进入发布流程 | 双重确认（内容 + 渠道） |
| **学术警告** | 检测到学术关键词 | 强制阅读 + 勾选确认（不可忽略） |

#### 学术警告强制确认流程

**触发条件**：检测到学术关键词（论文/毕业/期刊/SCI/第一作者等）

```
检测到学术关键词 → 显示强制确认弹窗 → 用户必须勾选"我已知晓" → 才能继续
```

**学术关键词库**（见 `references/keywords.md`）：
- 论文类：论文/毕业论文/学位论文/毕业设计
- 期刊类：期刊/投稿/发表/SCI/SSCI/CSSCI/核心期刊
- 学术类：学术/研究/科研/课题/基金项目/学术诚信
- 署名类：第一作者/通讯作者/署名
- 数据类：实验数据/调查数据/伪造数据/数据造假

## §3.X 全阶段产出物保存总则

> **根因**：用户确认章节后，Agent仅在对话中展示内容，未调用write_to_file保存，导致草稿丢失。  
> **规范**：所有阶段产出物必须立即保存。

### 产出物编号对照表

| 阶段 | 产出物 | 编号格式 | 文件路径 |
|------|--------|----------|----------|
| S0 | 调研简报 | `[S0] 调研简报` | `{workspace}/[S0] 调研简报.md` |
| S1 | 定位确认稿、风格档案、封面初稿 | `[S1] xxx` | `{workspace}/[S1] xxx.md` |
| S2 | 目录草案、视觉规划表 | `[S2] xxx` | `{workspace}/[S2] xxx.md` |
| S2.5 | 脉络确认稿 | `[S2.5] 脉络确认稿` | `{workspace}/[S2.5] 脉络确认稿.md` |
| **S3** | **第N章、质量报告** | **`[S3-ChNN]`** | **`{workspace}/[S3-ChNN] 第N章.md`** |
| S4 | 全书合稿、封面定稿、构建输出 | `[S4] xxx` | `{workspace}/[S4] xxx.md` |
| S5 | 终稿质量报告、数据保鲜表 | `[S5] xxx` | `{workspace}/[S5] xxx.md` |
| S6 | 完结回顾、运行日志、产出物总清单 | `[S6] xxx` | `{workspace}/[S6] xxx.md` |

### 强制保存检查点

```
用户确认产出 / 用户说"下一章" / 用户说"继续"
    ↓
【必须】立即调用 write_to_file 保存产出物
    ↓
文件路径：{workspace}/[产出物编号] 产出物名称.md
    ↓
保存成功后输出：✅ [产出物编号] 已保存
    ↓
才可进入下一阶段/下一章
```

### 失败处理规范

| 失败场景 | 处理流程 |
|---------|---------|
| 保存失败 | 重试最多3次，每次间隔5秒 |
| 3次仍失败 | 输出完整内容到对话 + 提示用户手动复制 |
| 写入验证失败 | 检查路径是否正确，尝试创建父目录 |

## S0 前置调研模块

> **位置**：S1之前，用户启动"写书"后**首个环节**  
> **触发**：用户说"写书"后**自动执行**，除非用户说"跳过调研直接写书"  
> **流程整合**：融合全球化调研、资产复用、变现策略等核心决策点（见各专项文档）

### 主题锁定机制（P0 强制）

**目的**：防止上下文跳变（如「龙虾」→「家谱」），确保整个 S0–S6 流程主题一致。

```
S0 启动时必须执行：
1. 【主题声明】向用户显式声明："当前写作主题：[用户提供的主题]"
2. 【主题记录】将主题写入调研简报首行："**主题**：[主题名称]"
3. 【主题锁定】锁定后，任何阶段切换必须携带主题校验
4. 【漂移检测】若检测到用户输入与锁定主题不一致，触发确认："您是要继续写[原主题]，还是切换到新主题[新输入]？"
```

**主题漂移阻断规则**：

- 用户说「确认」时，必须附带主题上下文：「确认继续写[主题]？」
- 读取任何工作记忆/历史文件前，必须先校验文件主题与当前主题一致
- 主题不一致时，禁止自动加载历史文件，必须显式询问用户

### 执行流程

**S0 = 选题验证 + 市场调研 + 风险评估**

```
S0执行流程（三项并行，约5-10分钟）：
  ├─ S0-A 竞品扫描（全球市场）
  ├─ S0-B 读者痛点分析
  └─ S0-C 变现潜力初判

【新增】S0-D 主题锁定与一致性声明
  ├─ 显式声明当前主题
  ├─ 写入调研简报首行
  └─ 建立主题校验基准
```

### 主体内涵锚定（P0，防 S0 漂移）

> **「竞品」「变现」在 S0 中的默认含义以「本书/本稿」为锚，不以「主题所在行业」偷换主体。**

| 术语 | **默认指**（须优先满足） | **易混项**（须拆分，不可顶替默认列） |
|------|--------------------------|--------------------------------------|
| **S0-A 竞品** | 与**本交付物**在同一议题下**可替代**的内容：同类**书/白皮书/手册/深度长文/付费专栏/课程化内容**等（读者若不读本稿，会改读的那些东西） | 主题产业链上的**商业产品/软件/Agent**（如某桌面 AI 产品矩阵）→ 仅当体裁为**工具评测、采购指南、产品横评**且用户确认时，可作为**并列维度**检索，并在简报中**分表**（「内容竞品」与「产品参照」勿混为一谈） |
| **S0-C 变现** | **本书/本稿**如何产生收入与触达读者：**版税/电子书与付费阅读/企业采购与定制/授权转载/训练营与咨询配套**等 | 主题赛道的**融资、行业规模、公司商业模式** → 仅作**市场上下文**小节，**不得**代替「本稿怎么卖、谁付钱」的结论 |

### S0-A 竞品扫描（内容替代物优先）

- 搜索维度（默认）：国内若干 + 海外若干 **同类出版物或等效长内容**（书名/报告标题/头部文章/课程大纲级竞品）
- 搜索语言：中文 + 英文（必要时日文/韩文）
- 输出：**内容竞品对照表**（定位、篇幅、渠道、差异点、空白位）+ 差异化机会；若存在产品横评体裁，**另表**「产品参照（非默认竞品）」
- **禁止**：在默认体裁下把「行业里的 App/Agent」当成 S0-A 的唯一竞品叙事而不提同类书/文。

### S0-B 读者痛点分析

- 基于用户输入的书籍主题，分析**会读这本/这篇的人**
- 搜索读者痛点、需求场景、**现有读物/服务**仍解决不了什么（与 S0-A 内容竞品形成闭环）
- 输出：读者画像 + 核心痛点列表（≥3条）

### S0-C 本书/本稿变现潜力初判

- 在 S0-B 读者与（默认）S0-A **内容竞品**基础上，评估 **本稿** 的可行变现与分发路径
- 可引用行业规模/趋势作**论据**，须单独小节标注为「市场背景」，**不得**用赛道变现叙事替代本稿变现结论
- 输出：建议的**本稿**变现与分发组合（如纸质+电子书、ToB 采购、专栏连载、配套课等）

### S0决策点

| 选项 | 执行 | 说明 |
|------|------|------|
| 默认 | 执行S0 | 自动执行三项调研，约5-10分钟 |
| 跳过 | "跳过调研直接写书" | S0置空，直接进入S1；**须向用户可见展示**跳过确认（例如：「已按您的要求跳过 S0，将无竞品/读者/变现简报；后续事实与数据仍须在 S1+ 阶段检索补齐」），并写入运行日志/阶段状态，**禁止静默跳过**。 |

## 阶段1：定位 + 风格 + 策略选择

**加载** → `presets.md` + `visual.md` §1封面 + `strategy.md` §1-§3

### S1 主题确认审批点（P0 强制）

**进入 S1 时必须执行**：

```
1. 【主题回显】"确认当前写作主题：[S0锁定的主题]"
2. 【一致性检查】若 S0 未锁定主题，禁止进入 S1，必须先完成 S0 主题锁定
3. 【用户确认】"请确认：我们要写的是关于[主题]的[体裁]，对吗？"
4. 【主题写入】将确认后的主题写入 [S1] 定位确认稿首行："**主题**：[主题]"
```

**主题变更处理**：

- 用户在 S1 阶段要求变更主题 → 触发 S0 重启流程
- 主题变更必须显式声明：「主题已从[旧主题]变更为[新主题]，将重新执行 S0 调研」

### 写作模式识别

#### 模式识别矩阵

| 用户输入特征 | 推荐模式 | 触发关键词示例 |
|-------------|---------|---------------|
| **家族史/通史** | 通史模式 | "XX万字，XX家族史"/"中国通史"/"世界通史"/"XX史" |
| **传记/回忆录** | 传记模式 | "写我爷爷"/"写我自己"/"回忆录"/"XX的一生" |
| **手册/指南** | 手册模式 | "产品手册"/"使用指南"/"教程"/"实战" |
| **白皮书/报告** | 白皮书模式 | "行业白皮书"/"研究报告"/"调研报告"/"XX分析" |
| **框架/大纲** | 框架模式 | "帮我写大纲"/"整理框架"/"生成目录"/"梳理结构" |
| **案例集** | 案例模式 | "案例分析"/"实战案例"/"项目案例"/"案例库" |

#### 识别流程

1. 用户输入 → 关键词匹配 → 匹配到关键词
2. 判定置信度：
   - 高置信度（>80%）：关键词明确匹配 → 自动采用该模式
   - 中置信度（50%-80%）：关键词模糊 → 推荐模式，询问确认
   - 低置信度（<50%）：无匹配关键词 → 提供模式选项
3. 执行对应模式策略

### G4学术警告扫描

**触发条件**：检测到学术关键词

```
G4扫描流程（不可跳过）：
1. 加载 references/keywords.md 关键词库
2. 扫描用户输入（核心主张/目标读者/书籍主题）
3. 匹配检测：
   - 强触发（弹窗+强制确认）：论文/毕业/期刊/署名/实验数据/诚信类关键词

```

**加载** → 按Task角色精准加载（见§5分阶段加载表）

### S3 启动与章前检索：防卡顿（体验强约束）

> **现象**：用户确认大纲后，界面长时间无新消息，误以为「卡住」——常见于 **章前检索串行**（每 query 可达策略超时，默认约 15s 量级）、**模型首轮批量 Read 多份长规范**、或 **工具执行中无可见状态**。

| 根因 | 对用户侧建议（模型/宿主须落实） |
|------|--------------------------------|
| 章前 ≥N 次检索依次执行 | **首条可见回复**须明示：已进入 **S3**、本章标题/编号、将先做 **章前检索 N 次**（N 以 `search-policy.json` 为准），再进入成稿；不得静默开检。 |
| 单次检索等待 | 每轮检索进行中：若连续 **>30s** 无可见产出，须按 `SKILL.md` §1 **心跳**约定 **≤15s** 一行状态（例如「章前检索 1/2…」）；宿主 UI 宜展示工具进行中。 |
| 规范冷读 | **在输出本章任何成稿段落之前**，避免无说明地并行 **Read 超过 3 份长文档**；优先走 [`skill-index.md`](./skill-index.md) 的短路径或会话已加载的节选。 |
| 检索失败/超时 | **立即**一句话说明原因与可重试项，禁止长时间空白后突然输出正文。 |

**集成侧（可选）**：使用 `BookWorkflowOrchestrator.execute({ mode: 'chapter_gate', …, onProgress })` 或 `chapter_pipeline` 时，章前门禁会对 **每次检索** 触发进度事件（见 `SearchBundle.executeChapterGate` / `MultiAgentPipeline`），宿主应将 `nextAction`、`gateQueryIndex` 等映射为可见文案，降低卡顿感。

### S3 多路并行写作（WorkBuddy 多 Writer）：依赖、台账与共享工件

> **审计背景**：并行时常见失败不是「章节没写」，而是 **状态不可见**、**无依赖顺序**、**引用格式各自为政**。本节为 **规范层** 要求；宿主无原生调度时由 **team-lead** 人工落实。

| 动作 | 要求 |
|------|------|
| **初始化** | 本书根执行 `node scripts/init-fbs-multiagent-artifacts.mjs --book-root <本书根>`，生成 `.fbs/chapter-status.md`、**本书根 `chapter-status.md`（同模板）**、`GLOSSARY.md`、`chapter-dependencies.json`、`book-context-brief.md`、`search-ledger.jsonl`、`project-config.json`；并跑 `node scripts/shared-knowledge-base.mjs --book-root <本书根>` 自检。 |
| **文件命名** | 章节落盘须与 §3.X 一致：**`[S3-ChNN] 第N章-标题.md`**；避免 `S2-草稿-` 等前缀造成进度误判。 |
| **依赖与批次** | S2 目录表增加 **`依赖章节`** / **`并行批次`** 列；**有向无环**：仅当依赖章在 `chapter-status.md` 中为「已完成」后，team-lead 再派发下游章。 |
| **完成通知** | 每章 Writer 完成后：**(1)** 更新 `chapter-status.md` 对应行；**(2)** 向 team-lead 发结构化摘要（章名、文件路径、字数、时间）；**(3)** 更新 `book-context-brief.md` 中本章「一句结论 / 末段钩子」表。 |
| **术语** | 并行启动前须有 **`.fbs/GLOSSARY.md` 或 S1/S2 术语表**；Writer 任务须引用路径（与 `book-level-consistency.md` C0-2 一致）。 |
| **引用格式** | 主编向全员广播 [`citation-format.md`](../02-quality/citation-format.md) 选定格式；**G4** 在并行模式下默认启用，见 `quality-check.md` §G。 |
| **检索门禁** | team-lead 合并前可对每章运行 `node integration/enforce-search-policy.mjs --skill-root <技能根> --chapter-file <路径>`。 |
| **依赖图与磁盘** | 维护 `.fbs/chapter-dependencies.json`；用 `node scripts/sync-book-chapter-index.mjs --book-root <本书根> --json-out .fbs/chapter-scan-result.json` 核对 **MEMORY 与文件是否一致**；`node scripts/chapter-scheduler-hint.mjs` 提示可派发章。 |
| **派发前依赖门禁** | `node integration/chapter-dependency-gate.mjs --book-root <本书根> --chapter <id>` — 退出码 **1** 表示依赖章稿件在磁盘上未匹配到，**禁止**开写（测试报告 01·TC-04）。 |
| **引用 / 术语** | `integration/citation-format-check.mjs`、`integration/terminology-gate.mjs`（可选 `--strict`）。 |
| **心跳 / 失败队列** | `integration/heartbeat-watchdog.mjs`（别名 **`heartbeat-monitor.mjs`**）、`integration/task-queue-helper.mjs`（别名 **`task-requeue.mjs`**）（测试报告 03·08）。 |
| **跨章审查** | 合稿后执行 **CX**（[`cross-chapter-consistency.md`](../02-quality/cross-chapter-consistency.md)），再进入 C0 / S5。 |
| **心跳与超时** | 协议见 [`../05-ops/heartbeat-protocol.md`](../05-ops/heartbeat-protocol.md)；长文档审查须 **抽样/分块**，避免单任务全量扫描导致静默超时（与 `workbuddy-agent-briefings.md` SOP 一致）。 |

### 置信度阈值审批机制

| 置信度（0–10） | 行为 | 用户感知 |
|--------|------|----------|
| **≥8.5** | 静默通过 | 实时标记"✅ AI已确认" |
| **6.0–<8.5** | ambient notification | 不打断，右下角toast提示"AI已校验，内容供参考" |
| **<6.0** | explicit approval | 触发审批弹窗"内容需人工确认，请检查以下章节..." |

### 每章执行流程

```
主Agent串行执行（心跳监控，§6.4）：
  → 联网搜索本章数据
  → 生成初稿
  → 生成视觉元素标记（Mermaid占位+插图标记）
  ↓
Critic-S 自动预过滤（S1-S6 + V1视觉密度）→ 自动修正
  ↓
四审并行启动（Task角色，可选降级为串行）：
  ├─ Critic-L1（初审）：P层4条逐段
  ├─ Critic-L2（复审）：C层4条逐章 + 知识准确性（多路并行时叠加 `quality-PLC.md` §C5 衔接）
  ├─ Critic-L3（终审）：B层5条篇级 + 视觉×内容相关性
  └─ Proofer（校对）：排版底线 + 数据一致 + 通读
  ↓
合并反馈 → Writer修改
  ├─ 评分≥7.5 → 输出给用户
  ├─ 7.0–<7.5 → 建议修订后再审
  └─ 评分<7.0 → 内部迭代（≤2轮）
      └─ 2轮后仍<7.0 → 标记问题交用户决策
  ↓
输出：章节内容 + 精简评分
```

### S4 排版构建：防卡顿（体验强约束）

> **现象**：进入 S4 后长时间无消息——常见于 **`node assets/build.mjs`**：依赖安装、**Chromium 冷启动**、`page.setContent` 使用 **`networkidle0`（最长约 90s）**、**Mermaid CDN 渲染等待（最长约 45s）**、大书 **DOCX 转换**；会话内若由模型代跑 Bash，用户看不到终端输出时更易误判「卡死」。

| 根因 | 模型 / 宿主须落实 |
|------|-------------------|
| 本地构建长静默 | 在发起 `node …/assets/build.mjs` **前**向用户说明：将经历「解析 MD → 写 HTML →（可选）PDF/DOCX」，PDF 阶段可能 **数十秒无新输出**；建议用户同时看终端或日志。 |
| 终端有输出但对话无 | `build.mjs` v2.2+ 会输出 **`[S4/build][书id]`** 前缀进度行；模型可将终端尾部 **摘要贴回对话**（每 **≤30s** 或每完成一书），与 `SKILL.md` §1 心跳一致。 |
| 规范冷读 | 进入 S4 **首轮**避免无说明地 **Read 超过 3 份长文档**；`typography` / `build.md` 按需 **节选** 或引用 skill-index 短链。 |
| 仅要 MD | 若用户只要成稿 MD，**不必**强跑 Puppeteer；避免为「完整构建」而拉长无反馈时间。 |

**代码侧**：`assets/build.mjs` 已在关键步骤打印 `[S4/build]` 进度；集成宿主可选解析该前缀驱动 UI 进度条。
