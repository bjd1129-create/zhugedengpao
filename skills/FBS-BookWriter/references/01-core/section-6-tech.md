## 6.3 环境预检协议

每个阶段开始前自动执行环境预检，确保所需能力可用：

### 预检清单

| 检查项 | 验证方式 | 失败处理 |
|--------|----------|----------|
| 联网能力 | 执行一次test搜索 | 降级为本地知识库 |
| Mermaid CDN | 检测unpkg/jsdelivr连通性 | 使用本地mermaid.js |
| 文件系统 | 测试写入权限 | 提示权限问题 |
| Task并行 | 检测team_create可用性 | 降级为串行执行 |
| 团队通信 | 检测send_message可用性 | 单Agent模式，禁用多成员协作 |
| 心跳机制 | 验证心跳发送/接收通道 | 降级为单Agent串行执行 |

### 预检执行时机

- S1开始前（定位阶段）
- S3开始前（写作阶段）
- S4开始前（构建阶段）

### 预检输出格式

```
🔍 环境预检
  联网搜索：✅ 可用
  Mermaid CDN：✅ unpkg可用
  文件写入：✅ 可写
  Task并行：✅ 可用
  团队通信：✅ 可用
  心跳机制：✅ 通道正常
  → 环境就绪，开始执行
```

### 环境预检与心跳监控的协同关系

- 预检通过 → 启用完整心跳监控
- 预检失败但可降级 → 降级模式运行（串行/单Agent）
- 预检失败无法降级 → 中断执行，提示用户修复

## 6.4 心跳监控与关闭协议

### 6.4.1 心跳监控协议

**监控主体**：team-lead 是心跳监控的唯一主体。所有团队成员的心跳统一发送给team-lead，由team-lead汇总状态后向用户播报。

**心跳频率**：每60秒一次

**心跳包格式**（JSON）：
```json
{
  "type": "heartbeat",
  "sender": "成员代号",
  "current_task": "当前执行的任务描述",
  "progress": 0-100,
  "status": "IDLE | WORKING | WAITING | COMPLETE",
  "timestamp": "2026-03-21T16:50:30Z"
}
```

**team-lead监控逻辑**：

| 场景 | 触发条件 | 执行动作 |
|------|---------|---------|
| 正常 | 收到心跳 | 更新成员状态记录 |
| 警告 | 120秒无心跳 | 发ambient notification警告用户 |
| 异常 | 300秒无心跳 | 强制shutdown该成员，记录异常 |

**超时响应流程**：

```
首次警告（120秒无心跳）:
  → 向用户发送ambient notification
  → 内容："[成员]已超过120秒未更新状态，可能存在阻塞"

强制shutdown（300秒无心跳）:
  → 调用shutdown_request到该成员
  → 记录异常到运行日志
  → 继续其他成员监控
```

**成员端职责**：
- 每个成员必须每60秒发送一次心跳
- 心跳内容应反映真实进度，不得虚假心跳
- 任务状态变化时立即发送状态变更通知

---

### 6.4.2 shutdown_response强制收齐协议

**原则**：team-lead调用shutdown时，必须等待**所有成员**返回shutdown_response后才执行team_delete。

**关闭流程**：
```
1. team-lead发送shutdown_request到所有活跃成员
2. 等待所有成员返回shutdown_response
3. 收到所有shutdown_response后，执行team_delete
```

**超时重试机制**：

| 轮次 | 等待时长 | 动作 |
|------|---------|------|
| 第1次 | 60秒 | 等待shutdown_response |
| 超时 | — | 重试发送shutdown_request（最多1次） |
| 第2次超时 | 再等30秒 | 强制删除，静默成员记录异常 |

**异常处理**：
- 静默成员（不响应shutdown）：强制删除并记录`ANOMALY: silent_member`
- 部分响应：已响应成员标记为`shutdown_ack`，未响应成员强制清理
- 网络抖动：重试1次，留15秒余地

**完整关闭序列示例**：

```
team-lead关闭团队时：
  → 发送shutdown_request到 [member-1, member-2, member-3]
  → 等待响应（60秒）
  → [member-1] 响应 ✓
  → [member-2] 响应 ✓
  → [member-3] 超时
  → 重试shutdown_request到 [member-3]（第2次）
  → [member-3] 响应 ✓
  → 执行 team_delete
  → 记录：团队正常关闭，member-3响应延迟80秒
```

## 6.5 可靠性设计原则（2026最佳实践）

**硬性限制**（防止Agent失控）：
- 每次运行最大工具调用次数：50次
- 每个工具最大重试次数：3次
- 每次运行最大总步骤数：100步
- 每次运行最大运行时间：10分钟

**故障分类与响应策略**：

| 故障类型 | 响应策略 |
|----------|----------|
| 瞬时故障（超时/限流） | 带退避重试（1s→2s→4s） |
| 确定性故障（格式错误） | 停止重试，修复输入 |
| 外部依赖故障 | 降级方案，通知用户 |
| 策略阻止 | 请求批准，上报人类 |

**降级机制**：
- Mermaid CDN失败 → 本地mermaid.js
- Task并行失败 → 串行执行
- AI图像失败 → L2程序化替代
- PDF生成失败 → 用户自助转换指南

**状态终端定义**：
- `NEED_INPUT`：需要用户提供信息
- `NEED_APPROVAL`：需要用户确认
- `QUEUED`：任务已加入队列
- `SAFE_FAILURE`：安全失败，已记录

### 6.5.1 宿主侧联网、心跳与写盘（交付审核约定）

> 本节供集成方对照 `integration/CHECKLIST.md` 验收；本技能包不实现具体 HTTP 客户端，由宿主落实超时与重试。

| 项 | 建议 |
|------|------|
| **联网检索超时** | 单次 WebSearch/WebFetch **30–60s** 上限；超时按「瞬时故障」退避重试（如 1s→2s→4s），仍失败则向用户展示摘要级失败原因，**不得**用模型记忆编造事实句顶替。 |
| **Writer 内循环** | 与上文「最大工具调用 / 最大步骤」一致；单章内部为提升分数的改写循环建议 **≤8 轮**或合并计入总步数上限，避免无限润色。集成参考实现中 **`agents.*` 单步默认 300s 超时**（`BookWorkflowOrchestrator` 的 `roleStepTimeoutMs`，**0** 关闭）。 |
| **心跳丢失** | 若 **连续 2 个**心跳周期（默认 60s/次，见 §6.4）未收到子成员心跳，宿主应暂停派发新工具调用、向用户播报卡死成员，并允许重试或降级串行。 |
| **写盘** | 产出物写入失败时 **最多 3 次**重试（间隔约 5s）；仍失败则输出全文到会话并提示手动保存（与 `section-3-workflow` 失败表一致）。 |
| **PDF/Mermaid** | 构建侧若启用 CDN Mermaid，导出 PDF 前须等待图表渲染完成（见 `assets/build.mjs` 内等待逻辑）。 |
