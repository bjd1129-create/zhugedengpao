## 设计原则

1. **分层处理**：复杂意图 → LLM推理，简单指令 → 规则匹配
2. **模糊容忍**：支持语音转文字错字、口语化表达、中英文混合
3. **降级保障**：LLM不可用时自动降级到规则匹配，不阻断用户操作
4. **意图优先**：先识别意图，再提取参数，避免过度解析

### 与触发词首响对齐（福帮手生产约定）

> 当宿主已路由到 **FBS-BookWriter** 且意图含 `WRITE_BOOK` / 「写书」等：不得用泛泛一句「你想写什么？」作为唯一回复。须遵守 [`SKILL.md`](../../SKILL.md) **「技能加载后的行为约定 → 触发词首响」**：无主题则 **S0 说明 + 合并收集**；有话术中主题则 **禁止重复追问主题**。
> 身份类问询（如“你是谁”）遵循 `SKILL.md` 的统一身份自述口径：以“福帮手 · FBS-BookWriter 人机协同写书技能”为主，不回退到泛化宿主介绍。

## 8.1 意图识别（Intent Recognition）

### 顶级意图池（共9个核心意图）

| 意图 | 中文触发 | 英文触发 | 映射流程 |
|------|----------|----------|----------|
| `WRITE_BOOK` | 写书/写手册/写指南/写白皮书 | write book/guide/paper | → S0前置调研 → S1 |
| `CONTINUE` | 继续写/接着来/往下写 | continue/go on | → 当前章节续写 |
| `REVIEW` | 检查质量/审一审/自检 | check/review quality | → Critic审核流程 |
| `EXPORT` | 导出/打包/下载/生成 | export/download/pack | → S4构建输出 |
| `STRATEGY` | 换策略/改深度/切换模式 | change strategy/mode | → S2策略选择 |
| `STATUS` | 进度如何/到哪了/状态；**卡住/没反应/还在吗/无响应/是不是挂了**（长任务心跳语境） | status/progress/still working/any update | → 运行状态快照 + **短时进度回显**（与 `SKILL.md` §0 长静默预告一致，不视为新意图） |
| `STOP` | 终止/停止/取消/中断/退出 | stop/cancel/exit | → 安全结束流水线，保存断点（若宿主支持），停止自动工具调用 |
| `HELP` | 怎么用/帮帮我/指令 | help/how to | → 指令引导（见 [`section-4-commands.md`](./section-4-commands.md) 系统控制类） |
| **`CONFIRM_TOPIC`** | **确认/是的/没错**（在需确认语境下） | **confirm/yes/proceed** | → **主题一致性校验** → 继续当前流程 |

### 意图识别示例

```
用户输入："帮我写一本关于AI的书"
↓
NLU解析：
{
  "intent": "WRITE_BOOK",
  "entities": {
    "topic": "AI",
    "type": "专业书籍"    ← 识别为"书籍"而非"手册/白皮书"
  },
  "confidence": 0.95,
  "fallback": false
}
↓
映射到S0流程

用户输入："检查一下质量"
↓
NLU解析：
{
  "intent": "REVIEW",
  "scope": "全章",        ← 识别为全章审查
  "detail_level": "精简"  ← 默认精简模式
}
↓
映射到Critic审核流程
```

## 8.1a 主题一致性校验（新增）

### 校验触发点

| 场景 | 校验动作 | 失败处理 |
|------|----------|----------|
| 用户说「确认」 | 检查当前是否有锁定主题；如有，回显「确认继续写[主题]？」 | 无锁定主题时，询问「确认写什么？」 |
| 读取工作记忆文件前 | 检查文件主题字段与当前主题一致 | 不一致时，询问「检测到历史项目[旧主题]，是否切换？」 |
| 新会话启动 | 检查是否有未完成的主题 | 有未完成主题时，询问「是否继续写[主题]？」 |
| S0→S1→S2→S3 阶段切换 | 携带主题字段，校验一致性 | 主题不一致时，阻断并询问 |

### 主题漂移检测算法

```
function detectTopicDrift(userInput, lockedTopic, confidenceThreshold=0.6):
    # 1. 提取用户输入中的主题信号
    signals = extractTopicSignals(userInput)

    # 2. 与锁定主题比对
    similarity = calculateSemanticSimilarity(signals, lockedTopic)

    # 3. 漂移判定
    if similarity < confidenceThreshold:
        return {
            "driftDetected": true,
            "lockedTopic": lockedTopic,
            "suggestedNewTopic": signals.primary,
            "action": "ASK_CONFIRMATION"
        }
    else:
        return {"driftDetected": false}
```

### 主题锁定状态机

```
[无主题] --用户说"写书"--> [S0: 主题待锁定]
[S0: 主题待锁定] --完成S0调研--> [S1: 主题已锁定]
[S1: 主题已锁定] --用户确认--> [S2+: 主题锁定中]
[S2+: 主题锁定中] --用户说"新主题"--> [主题变更确认]
[主题变更确认] --用户确认变更--> [S0: 主题待锁定] (重启)
[主题变更确认] --用户取消--> [S2+: 主题锁定中] (保持)
```

## 8.2 实体提取（Entity Extraction）

### 标准实体类型

| 实体 | 标识符 | 示例 | 提取规则 |
|------|--------|------|----------|
| 主题 | `{topic}` | "AI"、"企业数字化转型" | 名词短语优先 |
| 书名 | `{book_name}` | "《AI实战指南》" | 带书名号或引号 |
| 章节 | `{chapter}` | "第3章"、"第三章" | 正则：`第[一二三四]+章` |
| 关键词 | `{keyword}` | "AI绘画"、"LangChain" | 学术术语优先 |
| 数字 | `{number}` | "10万字"、"5章" | 阿拉伯数字+单位 |

### 实体验证规则

- 提取后需回显确认：`您是说写一本关于"{topic}"的书吗？`
- 模糊时触发确认：`"AI"是指人工智能还是...？`

## 8.3 模糊匹配规则

### 同义扩展池

| 标准指令 | 同义变体（中文） | 同义变体（英文/混合） |
|----------|------------------|----------------------|
| 写书 | 写书/写本书/写个书/要写书/想写书/帮我写书 | write a book/write book |
| 继续写 | 继续/接着/接着来/往下写/继续写/继续往下 | continue writing/go on |
| 检查质量 | 检查/审/审一审/检查质量/质量检查/自检 | check quality/review |
| 导出 | 导出/打包/下载/生成/输出 | export/download/output |
| 下一章 | 下一章/下一节/继续下一章 | next chapter |
| 封面 | 封面/换个封面/重新生成封面 | cover/cover image |
| 图表 | 图表/图/看看图/这个图 | chart/diagram |
| 策略 | 策略/换策略/改深度 | strategy mode |
| 进度 | 进度/到哪了/状态如何 | progress/status |
| 终止 | 停止/取消/中断/退出 | stop/cancel/exit |

### 模糊匹配算法

```
1. 预处理：去语气词（啊/嘛/呢）、统一标点、转小写
2. 规则匹配：检查是否命中同义扩展池（O(n)复杂度）
3. LLM推理：规则未命中时，使用LLM进行意图推理
4. 置信度计算：规则匹配=1.0，LLM推理=0.6-0.95
5. 降级阈值：置信度<0.5时触发"没听清，请再说一遍"
```

### 容错示例

```
"我想写本书" → 识别为WRITE_BOOK，提取topic=null待确认
"帮我检查一下" → 识别为REVIEW，scope=全章
"export" → 识别为EXPORT
"检查一哈质量" → "一哈"为语音错字，识别为REVIEW
"继续继续" → 识别为CONTINUE
"先停一下" / "终止" / "stop" → 识别为 STOP，触发安全收尾（见 `section-4-commands` 系统控制类）
```

## 8.5 NLU引擎选型

### 双层架构

```
┌─────────────────────────────────────────┐
│           用户输入                        │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│        规则匹配层（Rule Matcher）         │
│  - 同义扩展池查找                         │
│  - 正则实体提取                           │
│  - 响应时间：<50ms                        │
└─────────────────┬───────────────────────┘
                  ▼ 规则未命中
┌─────────────────────────────────────────┐
│       LLM推理层（LLM Intent Parser）       │
│  - 意图分类（9个核心意图 + OTHER 兜底）     │
│  - 实体提取                               │
│  - 置信度评估                             │
│  - 响应时间：500-2000ms                   │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│          置信度分流                       │
│  ≥0.85：直接执行                          │
│  0.6-0.85：执行+确认                      │
│  <0.6：询问澄清                           │
└─────────────────────────────────────────┘
```

### LLM提示词模板

```
你是FBS-BookWriter的NLU意图识别器。

用户输入：{user_input}

请识别：
1. 意图（必须与 §8.1 表一致；**仅允许以下取值**）：
   WRITE_BOOK | CONTINUE | REVIEW | EXPORT | STRATEGY | STATUS | STOP | HELP | CONFIRM_TOPIC | OTHER
   - STOP：终止/停止/取消/中断/退出
   - CONFIRM_TOPIC：在需确认语境下的「确认/是的/没错」等（主题一致性）
   - OTHER：无法归入以上九个命名意图时兜底，不得用 OTHER 代替明显的 STOP/CONFIRM_TOPIC
2. 关键实体（topic/chapter/scope等）
3. 置信度（0.0-1.0）

输出JSON格式。不要输出其他内容。
```

### 规则匹配降级时机

- LLM响应超时（>3秒）
- LLM返回格式错误
- 系统负载过高
- 网络不可用

## 8.7 意图确认机制

### 意图置信度分流

| 置信度 | 行为 | 示例 |
|--------|------|------|
| ≥0.85 | 直接执行，不打扰 | "写书" → 直接进入S0 |
| 0.6-0.85 | 执行+轻提示 | "写本...书？" → 执行+问"您是说写书吗？" |
| <0.6 | 先确认再执行 | "这个..." → "没听清，请再说一遍" |

### 参数确认示例

```
用户："帮我写本书"
NLU：intent=WRITE_BOOK, topic=null
系统：已进入S0前置调研（可说“跳过调研”）。请用一句话给出主题，可选补充读者与体裁。
用户：关于AI的
NLU：intent=WRITE_BOOK, topic="AI"
系统：已识别主题「AI」，开始执行 S0 前置调研并生成**内容竞品**（同类书/文/报告）/读者/**本书变现**简报（见 section-3-workflow「主体内涵锚定」）。
```

## 8.8 与现有§4短指令的衔接

### 兼容策略

- §4短指令作为规则匹配的底层实现
- §指令系统作为用户交互的前端接口
- 两者共享同一意图池和别名表

### 优先级

```
用户输入
  ↓
§指令系统 NLU解析
  ↓
┌─ 命中 §4 短指令关键字 → 执行对应动作
├─ 命中 NLU 意图 → 执行对应流程
└─ 未命中 → "没听清，请再说一遍"
```
