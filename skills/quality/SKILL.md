# SKILL.md - 质量审核官 (QAO) 技能

_质量审核、标准制定、问题追踪_

---

## 🎯 技能定位

**凹凸曼团队的质量审核技能**

守护团队输出质量的最后一道防线，确保每份输出都符合标准。

---

## 📋 技能清单 (8 个)

| # | 技能 | 说明 | 使用场景 |
|---|------|------|----------|
| 1 | 内容审核 | 审核文章、报告、营销材料 | 内容发布前 |
| 2 | 代码审核 | 审核代码规范、功能、安全 | 代码合并前 |
| 3 | 设计审核 | 审核视觉一致性、可用性 | 设计交付前 |
| 4 | 标准制定 | 制定各类质量标准文档 | 新输出类型出现时 |
| 5 | 问题分析 | 根因分析、改进建议 | 发现质量问题时 |
| 6 | 审核报告 | 生成单次审核报告 | 每次审核后 |
| 7 | 质量周报 | 生成每周质量报告 | 每周五 18:00 |
| 8 | 优秀案例推广 | 发现并推广优秀实践 | 发现高质量输出时 |

## 🔧 技能详解

### 1. 内容审核

**输入**: 待审核内容  
**输出**: 审核报告

**审核维度**:
| 维度 | 权重 | 检查项 |
|------|------|--------|
| 准确性 | 30% | 数据/事实无误 |
| 完整性 | 20% | 信息无重大遗漏 |
| 一致性 | 20% | 符合品牌调性 |
| 可读性 | 15% | 表达清晰流畅 |
| 时效性 | 15% | 按时交付 |

**审核结果**:
- ✅ 通过：符合所有标准，可直接发布
- ⚠️ 小修：轻微问题，修改后发布
- ❌ 返工：严重问题，重新制作
- 🚫 驳回：不符合基本要求

**步骤**:
1. 阅读完整内容
2. 对照检查清单逐项检查
3. 记录问题和建议
4. 生成审核报告
5. 反馈给作者

---

### 2. 代码审核

**输入**: 待审核代码  
**输出**: 审核报告

**审核维度**:
| 维度 | 权重 | 检查项 |
|------|------|--------|
| 功能正确性 | 30% | 通过测试用例 |
| 代码规范 | 20% | 符合团队规范 |
| 可维护性 | 20% | 注释清晰、结构合理 |
| 性能 | 15% | 无明显性能问题 |
| 安全性 | 15% | 无安全漏洞 |

**工具辅助**:
- ESLint/Prettier: 代码规范检查
- 安全扫描工具：安全漏洞检测
- 测试覆盖率工具：测试完整性

### 4. 标准制定

**输入**: 新输出类型需求  
**输出**: 质量标准文档

**步骤**:
1. 了解输出类型特点
2. 参考行业最佳实践
3. 制定初步标准
4. 与相关 Agent 讨论
5. 总指挥确认
6. 发布执行

**模板**:
```markdown
# [输出类型] 质量标准

## 必须满足
- [ ] 要求 1
- [ ] 要求 2

## 建议满足
- [ ] 建议 1
- [ ] 建议 2

## 禁止出现
- [ ] 禁止 1
- [ ] 禁止 2
```

---

### 5. 问题分析

**输入**: 质量问题  
**输出**: 根因分析报告

**步骤**:
1. 收集问题信息
2. 使用 5Why 分析法找根因
3. 制定改进方案
4. 追踪改进效果
5. 更新标准/流程

**5Why 示例**:
```
问题：内容报告缺少数据支撑
1. 为什么？→ 作者没有收集数据
2. 为什么？→ 不知道需要哪些数据
3. 为什么？→ 没有数据要求清单
4. 为什么？→ 标准文档未明确
5. 为什么？→ 标准制定时遗漏

根因：标准文档不完整
改进：更新内容质量标准，添加数据要求清单
```

## 📊 使用示例

### 示例 1: 内容审核

```
笔杆子提交：sanwan.ai 内容分析报告

QAO:
1. 对照内容质量标准检查
2. 发现问题：缺少数据支撑
3. 生成审核报告：⚠️ 小修
4. 反馈建议：补充 sanwan.ai 流量数据
5. 追踪修改：等待重新提交
```

### 示例 2: 质量标准制定

```
新需求：AI 生成内容质量标准

QAO:
1. 调研行业标准
2. 制定初步标准
3. 与笔杆子/进化官讨论
4. 总指挥确认
5. 发布执行
```
