# 官网运营全员工作流 v2.0

> 制定日期：2026-03-31 | 制定人：小花 | 升级：基于克隆审查教训

---

## 核心理念

**内容真实性第一** — 官网是老庄的数字化身，每一行字都必须真实。

```
铁律：禁止虚构内容
- 所有故事/数据/场景必须基于老庄真实经历
- 发现克隆内容立即清理，不得拖延
- 内容来源不明 = 不能上线
```

---

## 一、官网运营全景图

官网运营拆成4条流水线，同时进行：

| 流水线 | 负责人 | 输入 | 输出 |
|--------|--------|------|------|
| 内容流 | 文案君 | 老庄真实故事/经历 | 页面文案/博客文章 |
| 视觉流 | 配色师 | 页面需求/文案内容 | 配图/banner/图标 |
| 技术流 | 代码侠 | 设计稿/功能需求 | 页面开发/部署/性能 |
| 数据流 | 洞察者 | 竞品/用户/健康数据 | 优化建议/巡检报告 |

**流水线依赖关系：**
- 内容流和视觉流可并行
- 技术流依赖内容+视觉（先有内容再开发）
- 数据流独立运行，持续巡检

## 二、官网内容真实性检查机制

### 2.1 上线前检查（每次部署前必须执行）

**文案君 → 配色师 → 代码侠 → 部署**

每条流水线交付时必须附上：

```markdown
### 内容真实性声明
- [ ] 故事来源：老庄口述/日记/真实经历（附证据）
- [ ] 数据来源：真实统计/可核实（附来源）
- [ ] 无克隆内容：无抄袭 sanwan.ai / 三万 / 傅盛 等外部内容
- [ ] 品牌一致性：老庄/小花/ Dengpao品牌，不是其他品牌
```

**任何一条未通过 → 不得上线**

### 2.2 定期巡检（Cron + 协调官）

**每日巡检（洞察者 Cron）：**
```
检查项：
1. 首页内容是否仍为老庄/小花品牌
2. 是否有新增克隆内容（grep "傅盛/三万/sanwan/EasyClaw"）
3. 页面加载是否正常
4. 部署版本 vs GitHub 是否一致
```

**每周巡检（协调官）：**
```
1. 审查所有页面内容来源
2. 抽查3个页面的内容真实性
3. 检查外链是否指向正确品牌
4. 更新健康度报告
```

### 2.3 克隆内容快速处理

**发现克隆内容时：**
1. 立即标记"需清理"（不要等到周巡检）
2. 评估：删除 vs 重写
3. 脏页面 ≤ 2个 → 协调官直接处理
4. 脏页面 > 2个 → 升级小花 + 通知老庄

**克隆内容判断标准：**
```
出现以下关键词且与老庄无关 → 克隆内容：
- "傅盛" / "三万" / "sanwan" / "EasyClaw"（非我们品牌的）
- 骨折故事（傅盛经历，非老庄）
- "零人类内容"/"AI Agent三万"等外部项目描述
```

## 三、官网迭代标准流程（每个优化周期）

### 第1步：数据驱动发现问题（洞察者）

**触发：** 每周一 Cron 自动巡检 / 协调官手动触发 / 老庄指令

**执行内容：**
```
1. Lighthouse性能检测（Performance/SEO/Accessibility）
2. 竞品对比分析（sanwan.ai / 同类竞品）
3. 内容健康度扫描（克隆内容/外链/品牌一致性）
4. 流量和用户行为（如果GA可接入）
```

**输出：**
- `agents/researcher/memory/YYYY-MM-DD.md`（巡检报告）
- 发现问题 → 立即写入 `TASKS.md` 待处理项
- 克隆内容 → 标记"需清理"，同步通知协调官

---

### 第2步：问题立项与优先级（协调官）

**参与：** 协调官 + 小花（重大事项）

**执行内容：**
```
1. 召集数据review（洞察者汇报）
2. 确定本轮优化重点（最多3个P0-P1）
3. 拆解任务到4条流水线
4. 分配负责人和截止时间
5. 写入 TASKS.md
```

**优先级决策规则：**

| 优先级 | 类型 | 示例 | 决策权 |
|--------|------|------|--------|
| P0 | 影响转化 | 首页标题/CTA/核心页面 | 老庄确认 |
| P1 | 影响SEO/品牌 | 克隆内容/外链错误/meta | 协调官处理 |
| P2 | 影响体验 | 配图/banner/排版 | 配色师决策 |
| P3 | 锦上添花 | 动画/特效 | 视情况 |

---

### 第3步：内容先行（文案君）

**触发：** 协调官派发任务

**标准动作：**
```
1. 收到任务 → 5分钟内回复"收到" + 写入memory
2. 收集老庄真实素材（口述/日记/已有文字）
3. 撰写文案初稿
4. 完成时更新TASKS.md状态 + 提交审核
```

**交付标准：**
```
✅ 每篇文案必须基于老庄真实经历（禁止瞎编）
✅ 有具体数字/场景/情感细节
✅ 配图需求标注清楚
✅ 无克隆内容（自查：grep "傅盛/三万/sanwan"）
✅ 附内容真实性声明
```

**禁止：**
```
❌ 基于模板/泛泛而谈的内容
❌ 从其他网站复制粘贴的故事
❌ 没有具体细节的"大概"描述
```

---

### 第4步：视觉跟进（配色师）

**触发：** 文案君完成内容后自动启动 OR 并行

**标准动作：**
```
1. 收到任务 → 记录到memory
2. 审查现有页面配图 → 标注缺图页面
3. 设计配图方案 → 标注风格和用途
4. 交付图片 → 标注文件名和适用位置
```

**交付标准：**
```
✅ 配图风格：温暖、真实、龙虾吉祥物（xihua.jpg）
✅ 图片格式：WebP优先，单张<200KB
✅ 必须标注：图片用在哪个页面哪个位置
✅ 无来源不明的配图（自查版权）
```

---

### 第5步：技术落地（代码侠）

**触发：** 视觉流完成 OR 设计稿确认

**标准动作：**
```
1. 接收设计稿 → 评估实现难度
2. 技术实现 → 本地测试
3. Lighthouse复测 → 必须达标才部署
4. 部署上线 → 更新部署记录
```

**质量门槛（必须全部达标才能上线）：**

| 指标 | 最低标准 | 达标怎么办 |
|------|---------|-----------|
| Performance | ≥70 | 继续优化或接受现状 |
| SEO | ≥90 | 必须达标 |
| Accessibility | ≥90 | 必须达标 |
| 页面加载 | <3s | 必须达标 |

**部署前检查清单：**
```
✅ Lighthouse达标
✅ 无克隆内容（git diff 检查）
✅ 外链指向正确
✅ 内容真实性声明已附
```

---

### 第6步：数据验证（洞察者）

**触发：** 优化上线后7天

**执行内容：**
```
1. 对比优化前后的核心指标
2. 判断优化是否有效
3. 写入巡检报告
4. 有效 → 记录最佳实践
5. 无效 → 分析原因 → 进入下一轮优化
```

## 四、任务流转规则

### 4.1 传达链

```
老庄/小花 → 协调官 → 各Agent（主轨）
老庄/小花 → 各Agent（备轨，协调官失联时）
```

**失联判定（满足任一即触发备轨）：**
- 协调官超过30分钟无响应
- TASKS.md状态超过1小时未更新
- 任务超期24小时无进展
- 老庄明确指令

### 4.2 任务格式

```markdown
## [日期] 官网优化-xxx
- **类型**：内容/视觉/技术/数据
- **负责人**：@xxx
- **上游依赖**：xxx（如果有）
- **截止**：YYYY-MM-DD HH:MM
- **最小闭环动作**：完成xxx就算闭环
- **验收人**：协调官/小花/老庄
- **内容真实性声明**：✅已确认/⚠️待确认
- **状态**：待确认
```

### 4.3 状态定义

| 状态 | 含义 | 下一步 |
|------|------|--------|
| 待确认 | 已录入，等待负责人回复 | 负责人回复"收到" |
| 执行中 | 负责人回复"收到"并开始执行 | 完成或阻塞升级 |
| 待验收 | 负责人说完成，等待验收 | 验收人确认 |
| 已验收 | 验收人确认通过 | 协调官复盘 |
| 已闭环 | 全流程完成，已复盘 | 记录最佳实践 |

### 4.4 备轨触发动作

1. 小花直接在子Agent工作空间写入 `TODO.md`
2. 小花直接用飞书/sessions_send通知子Agent
3. 事后告知协调官（不追究，补记录即可）

## 五、各角色职责边界

### 协调官
- 发起数据review
- 拆解任务分配到各流水线
- 监控进度，推动闭环
- **重点：** 发现克隆内容立即处理，不超过24小时
- 重大事项上报小花/老庄

### 文案君
- 只做内容创作，不管设计和代码
- 交付物：文案初稿 + 内容真实性声明
- 配图需求告诉配色师，不要自己画

### 配色师
- 只做视觉设计，不管代码实现
- 交付物：设计稿/图片 + 文件位置标注
- 技术需求告诉代码侠，不要自己写代码

### 代码侠
- 只做技术实现，不管内容创作
- 交付物：可上线的页面
- 拒绝：无设计稿的"随便做一下"

### 洞察者
- 提供数据支撑，不做执行决策
- 交付物：数据分析报告 + 优化建议
- 建议必须有数据支撑，拒绝"我觉得"

## 六、闭环标准

**一个优化周期才算闭环，必须满足：**

```
1. 所有任务状态 = 已验收
2. 克隆内容检查通过（无脏页面）
3. 内容真实性声明已附
4. 洞察者完成数据验证（或标注跳过）
5. 协调官写复盘（what/wow/next）
6. 写入 memory/YYYY-MM-DD.md
```

**闭环复盘模板：**

```markdown
## 官网优化复盘 — YYYY-MM-DD

### 完成情况
- 优化了X个页面
- 清理了X个脏页面（克隆内容）
- 性能变化：Performance XX→XX

### 内容真实性检查
- [ ] 所有上线内容已核实来源
- [ ] 无克隆内容残留
- [ ] 品牌一致性通过

### 做得好的
- xxx

### 需要改进的
- xxx

### 下一步
- xxx
```

## 七、克隆内容快速响应（5分钟闭环）

**发现脏页面时，执行：**

```markdown
1. 立即标记：写入 TASKS.md "脏页面：xxx.html，需清理"
2. 评估处理方式：
   - 整页克隆（内容90%相同）→ 删除文件
   - 部分克隆（有我们内容但混了外部）→ 标注需要重写的段落
3. 脏页面 ≤ 2个 → 协调官直接处理，不上报
4. 脏页面 > 2个 → 升级小花 + 通知老庄
5. 处理完后更新 TASKS.md
```
